## 一、Summary（概要）

本计划对现有合同管理模块进行系统性重构与升级，引入通用工作流引擎、OnlyOffice 文档在线协作、文档版本控制、操作留痕、精细化权限审计六大能力，同时纠正现有合同模块未遵循 Controller-Service 分层、未使用 CI4 Migration、字段命名不一致等工程问题。

**核心交付物：**
1. 6 份设计文档（需求规格 / 用户流程图 / 系统架构 / 数据库设计 / API 接口设计 / OnlyOffice 集成方案）
2. 通用工作流引擎（表单式配置器，支持串行/会签/或签/抄送/加签/转签）
3. 合同模块重构（Controller-Service 分层 + 接入工作流引擎 + 列表筛选补全）
4. OnlyOffice 应用层集成（JWT 签名、回调保存、版本管理、操作留痕）
5. 权限审计扩展（权限变更审计表 + 审批人指派校验 + 按钮/数据级权限）
6. 数据迁移脚本（旧 `def_contract_master`/`def_contract_flow` → 新架构）
7. 测试套件（单元/集成/UAT）+ 用户操作手册 + 技术文档

### 2.3 权限系统现状

### 2.4 工程惯例（来自项目记忆）

- 后端 Controller-Service 分层，Service 按子域拆分，单文件 < 30KB
- `ContextService` 缓存 TTL 1800s，权限变更主动失效
- `RecordEditService`/`BatchEditService` 写操作后调用 `invalidateConfigCache()`
- 缓存需存储数据 + 表指纹（`information_schema.TABLES.UPDATE_TIME`）
- API 响应需包含 `X-Server-Trace` 头（分段耗时，SQL 内容 base64 编码，仅 >50ms 慢查询）
- 性能计时用 `hrtime(true)`，纳秒精度，`/1e6` 转毫秒
- `buildPerformanceTable` 集中在 `BaseApiController`
- ag-grid 数值列需显式 `cellStyle: { textAlign: 'right' }` + `agNumberColumnFilter` + 数值比较器
- pnpm 11 需 `.npmrc` 设 `verify-deps-before-run=false`，workspace `allowBuilds`
- 文本文件 LF 行尾，Windows 脚本 CRLF；`.gitattributes` 强制
- Git 配置 `core.autocrlf=false`/`core.eol=lf`/`core.filemode=false`

## 三、Assumptions & Decisions（假设与决策）

### 3.1 用户决策（Phase 2 已确认）

| 决策项 | 选择 |
|---|---|
| OnlyOffice 部署 | 假设 Document Server 已就绪，仅做应用层集成 |
| 审批流引擎范围 | 通用工作流引擎（合同作为首个接入方，未来可复用） |
| 实施策略 | 一次性全做（覆盖所有功能与文档） |
| 工作流前端复杂度 | 表单式配置器（串行/会签/或签/抄送/加签/转签，不引入 BPMN 拖拽） |
| 数据迁移策略 | 重构替换（新表 + 新 Service + 新引擎，旧数据迁移脚本导入） |

### 3.2 架构决策（基于项目惯例与探索结果）

| 决策项 | 选择 | 依据 |
|---|---|---|
| 后端分层 | Controller-Service 分层 | 与 Workbench/Employee 模块惯例对齐 |
| 数据库变更管理 | CI4 Migration（`app/Database/Migrations/`） | 纠正现有手工 SQL 方式，版本化追踪 |
| 工作流引擎独立 | `app/Services/Workflow/` 独立子域 | 通用复用，合同模块通过 facade 调用 |
| OnlyOffice 鉴权 | JWT（HS256）签名 + 回调验签 | OnlyOffice 官方推荐 |
| 文档版本策略 | 每次回调 `forceSave` 触发版本快照 + 手动版本标记 | 平衡存储与可追溯 |
| 权限审计 | 新增 `def_permission_audit` 表 + 扩展 `sys_sql_log` | 现有 `sys_sql_log` 为通用日志，权限变更需结构化审计 |
| 审批人指派 | 角色指派 + 部门指派 + 上级指派 + 指定人指派 | 覆盖常见企业审批场景 |
| 前端编辑器组件 | 通用 `OnlyOfficeEditor.vue` + 合同专用 `ContractDocEditor.vue` | 复用性与业务封装分离 |
| 字段命名 | 新表统一使用中文字段名（与项目惯例一致） | 与 `def_user`/`def_function` 等表风格一致 |
| 旧表处理 | 保留 `def_contract_master`/`def_contract_flow` 原样，迁移后归档 | 数据安全，可回滚 |

### 3.3 关键假设

1. OnlyOffice Document Server 访问地址通过 `.env` 的 `ONLYOFFICE_URL` 配置，JWT 密钥通过 `ONLYOFFICE_JWT_SECRET` 配置
2. OnlyOffice 回调地址需后端公网可达（或与 Document Server 网络互通），回调路径 `/onlyoffice/callback`
3. 现有 `def_contract_master` 历史数据量较小（< 10000 条），迁移脚本可一次性执行
4. 工作流引擎不引入 BPMN 引擎（如 Camunda/Flowable），自研轻量级表单式引擎
5. OnlyOffice 协同编辑并发冲突由 Document Server 内部处理，应用层仅处理版本快照
6. 现有 `view_role` 8 个操作权限不改动，新增工作流相关权限通过 `def_function` + `view_role.功能编码赋权` 扩展
7. 现有 `def_contract_type` 字典表保留复用

#### 3.3 关键逻辑

- **性能**：回调处理用 `hrtime(true)` 计时，写入 `def_doc_callback_log`

## 六、Risk & Mitigation（风险与缓解）

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| OnlyOffice Document Server 未实际部署 | 阶段 5/9 无法验证 | 阶段 1 文档明确部署要求，阶段 5 用 mock 回调先行开发，部署后联调 |
| 工作流引擎复杂度超预期 | 阶段 3 延期 | 表单式配置器（非 BPMN），自研轻量引擎，控制复杂度 |
| 旧数据迁移字段映射不全 | 阶段 2 数据丢失 | 迁移脚本先 dry-run，旧表保留归档，可回滚 |
| ag-grid 数值列对齐问题 | UI 显示异常 | 遵循项目惯例：`cellStyle: { textAlign: 'right' }` + `agNumberColumnFilter` + 数值比较器 |
| pnpm 11 严格依赖隔离 | 前端构建失败 | 显式声明 devDependencies，`.npmrc` 设 `verify-deps-before-run=false` |
| X-Server-Trace 头溢出 | 响应头截断 | 仅记录 >50ms 慢查询，SQL 内容 base64 编码 |
| 缓存与表结构变更不一致 | 数据不一致 | 写操作后 `invalidateConfigCache()`，缓存存储表指纹校验 |
| 行尾符问题 | Git 提交混乱 | `.gitattributes` 强制 LF，Windows 脚本 CRLF |
