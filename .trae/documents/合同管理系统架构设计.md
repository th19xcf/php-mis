## 1. 引言

### 1.1 目的

本文档对 MIS 系统合同管理模块的系统性升级进行架构层面的完整约定，明确模块划分、分层结构、子域依赖、关键流程、集成方式、安全与性能策略，作为后续数据库设计、API 接口设计、编码实现、测试验收与部署运维的统一架构依据。

本次升级在纠正现有合同模块工程问题（未遵循 Controller-Service 分层、未使用 CI4 Migration、`list()` 未实现筛选、`approve()` 未校验审批权限、字段命名不一致、`sprintf` SQL 拼接风险）的基础上，引入四大子域能力：

1. **Workflow 子域**：通用工作流引擎，支持串行/会签/或签/抄送/加签/转签；
2. **Contract 子域**：合同模块重构，接入工作流引擎与 OnlyOffice 文档协作；
3. **OnlyOffice 子域**：文档在线协作、版本控制、操作留痕；
4. **Permission 子域**：权限变更审计、按钮级与数据级权限补全。

### 1.2 范围

| 范围内 | 范围外 |
|---|---|
| 后端 Controller-Service 分层架构、子域划分、类设计 | OnlyOffice Document Server 本身的部署与运维 |
| 前端 View-Store-API 三层架构、模块划分、组件复用 | BPMN 可视化拖拽设计器（采用表单式配置器） |
| 工作流引擎核心架构与扩展点 | 工作流引擎接入其他业务模块（本次仅合同接入） |
| OnlyOffice JWT 双轨制、回调、版本、留痕 | 移动端适配（仅 PC 端） |
| RBAC + 按钮/数据级权限与审计 | Workbench/Employee 等现有模块改造 |
| CI4 Migration 版本化数据库变更 | 数字签名、电子签章（仅在线编辑与版本管理） |

### 1.3 设计原则

| 原则 | 含义 | 落地措施 |
|---|---|---|
| 高内聚低耦合 | 子域内部紧密聚合，子域之间通过明确接口通信 | 四大子域物理隔离（独立目录），通过 Service facade 调用 |
| 复用 | 通用能力下沉为基础设施，业务模块复用 | Workflow 引擎与 OnlyOffice 组件设计为可复用基础设施 |
| 可扩展 | 新业务模块接入工作流/文档协作无需修改引擎 | 通过回调接口 `onTaskCompleted()` 与通用编辑器组件扩展 |
| 安全 | 鉴权、防注入、防篡改、审计全覆盖 | JWT 双 Token + RBAC + 参数化查询 + 操作留痕 + 权限审计 |
| 性能 | 分段计时、缓存、索引、虚拟滚动 | hrtime 计时 + X-Server-Trace + MetadataCache + ag-grid 虚拟滚动 |
| 一致性 | 新代码遵循项目既有惯例 | 与 Workbench/Employee 模块分层一致、中文字段名一致、性能惯例一致 |

**分层职责**：

| 层 | 职责 | 约束 |
|---|---|---|
| Controller | 参数校验、调用 Service、响应封装、X-Server-Trace 输出 | 方法数 ≤ 10，不写业务逻辑，不直接操作 DB |
| Service | 业务编排、事务管理、调用 Library 与 Model | 单文件 < 30KB，按子域拆分 |
| Library | 无状态算法、配置构建、JWT、权限求值 | 不持有状态，可被多 Service 复用 |
| Model | 数据访问、表映射、SQL 执行 | 通过 Mcommon 执行参数化 SQL |
| DB | 持久化 | 中文字段名，CI4 Migration 管理 |

### 3.2 四大子域划分与依赖

```mermaid

        BaseApi[BaseApiController]

        AuthModel[AuthModel]
    end

    Contract -->|提交审批 启动实例| Workflow
    Contract -->|文档编辑 配置获取| OnlyOffice
    Contract -->|行级权限| AuthLib
    Workflow -->|审批人解析 角色查询| AuthLib
    OnlyOffice -->|权限映射 文档归属| AuthLib
    Permission -->|审计埋点 监听变更| AuthModel
    Permission -->|审计查询| AuthLib

    Contract -.继承.-> BaseApi
    Workflow -.继承.-> BaseApi
    OnlyOffice -.继承.-> BaseApi
    Permission -.继承.-> BaseApi
    Contract -.SQL.-> Mcommon
    Workflow -.SQL.-> Mcommon
    OnlyOffice -.SQL.-> Mcommon
```

**依赖规则**：

- Contract 子域依赖 Workflow 子域（提交审批、审批回调）与 OnlyOffice 子域（文档配置、版本查询）；
- Workflow 与 OnlyOffice 子域**互不依赖**，彼此独立；
- Permission 子域**不依赖**任何业务子域，仅监听公共权限模型的变更；
- 四个子域均依赖公共基础设施（BaseApiController、Mcommon、AuthorizationService、MetadataCache、AuthModel）；
- 严禁反向依赖：Workflow/OnlyOffice/Permission 不得调用 Contract Service。

**解耦设计要点**：

1. **回调接口契约**：业务模块实现 `BusinessCallbackInterface`，包含 `onTaskCompleted(instanceId, node, result)`、`onInstanceCompleted(instanceId, result)` 两个方法；
2. **回调注册表**：`BusinessCallbackRegistry` 按 `businessType` 路由回调，业务模块在应用启动时注册；
3. **Workflow 子域零业务知识**：引擎不感知"合同"概念，仅处理通用节点/任务/连线；
4. **业务模块主动调用**：`ContractWorkflowService::submit()` 主动调用 `WorkflowInstanceService::start()`，传入 `businessType=contract` 与 `businessId=合同ID`。

**双轨制要点**：

| 维度 | 用户 JWT | OnlyOffice JWT |
|---|---|---|
| 用途 | 用户身份鉴权、业务接口访问 | 文档配置签名、回调验签 |
| 密钥 | 系统用户密钥（`JWT_SECRET`） | `ONLYOFFICE_JWT_SECRET`（与 Document Server 一致） |
| 签发方 | 后端登录接口 | 后端 `OnlyOfficeJwtService::sign()` |
| 验签方 | 后端 `JwtAuthFilter` | 后端 `OnlyOfficeJwtService::verify()` + Document Server |
| 携带方式 | `Authorization: Bearer` 头 | `config.token` 字段 + 回调 `Authorization` 头 |
| 黑名单 | `TokenBlacklistService` | 无（短期有效，密钥轮换） |
| 过期 | accessToken 短期 + refreshToken 长期 | 编辑会话期间有效 |

**隔离原则**：两套 JWT 物理隔离，密钥不同、签发方不同、验签方不同，互不干扰。

**关键设计**：

1. **幂等性**：`def_doc_callback_log` 的 `(文档Key, status, forcesavetype)` 联合唯一索引 + 处理前预检，保证重复回调不重复创建版本；

**RBAC 关键规则**：

**映射实现**：

```php
// AuthModel::getUserAuthData() 扩展
public function getUserAuthData(int $userId): array
{
    $roles = $this->getUserRoles($userId);
    $functions = $this->getAuthorizedFunctions($roles);
    $buttons = $this->buildOperationButtons($roles, $functions);
    return [
        'roles' => $roles,
        'functions' => $functions,
        'buttons' => $buttons, // ['contract:add', 'contract:edit', ...]
    ];
}

protected function buildOperationButtons(array $roles, array $functions): array
{
    $buttons = [];
    $opMap = ['增' => 'add', '改' => 'edit', '删' => 'delete',
              '导入' => 'import', '导出' => 'export', '维护' => 'approve',
              '整表' => 'archive', '备注' => 'sign'];
    foreach ($functions as $func) {
        foreach ($roles as $role) {
            if (! in_array($func['功能编码'], $role['功能编码赋权'], true)) continue;
            foreach ($opMap as $field => $op) {
                if (! empty($role[$field])) {
                    $buttons[] = "{$func['功能编码']}:{$op}";
                }
            }
        }
    }
    return array_values(array_unique($buttons));
}
```

### 8.3 缓存策略

| 缓存对象 | 缓存键 | TTL | 表指纹 | 失效时机 |
|---|---|---|---|---|
| 流程定义 | `workflow:definition:{code}` | 1800s | `def_workflow_definition.UPDATE_TIME` | 流程定义保存/启用/停用后 `invalidateConfigCache()` |
| 合同主表 | `contract:master:{id}` | 1800s | `def_contract.UPDATE_TIME` | 合同 CRUD 后失效 |
| 文档配置 | `onlyoffice:doc:{docId}` | 1800s | `def_contract_doc.UPDATE_TIME` | 版本创建/回溯/标记后失效 |

**表指纹校验机制**：

1. 缓存写入时，同时存储 `information_schema.TABLES.UPDATE_TIME`（表最后修改时间）；
2. 缓存读取时，先查询当前表的 `UPDATE_TIME`，与缓存中的指纹比对；
3. 指纹不匹配则视为缓存失效，回源查询并刷新缓存；
4. 此机制保证即使忘记主动失效，表结构变更后缓存自动失效。

**实现约束**：

- 所有缓存读写通过 `MetadataCache` 统一组件，禁止直接调用 CI4 Cache；
- 写操作（INSERT/UPDATE/DELETE）后必须调用 `invalidateConfigCache($key)`；
- 缓存值必须可序列化（避免存储 PDOStatement 等资源）。

### 9.1 RESTful 设计规范

| 规范项 | 约定 | 示例 |
|---|---|---|
| URL 风格 | 短横线小写，资源名复数 | `/contract/list`、`/workflow/instance/detail/123` |
| HTTP 方法 | GET 查询、POST 创建/操作、PUT 更新（本项目以 POST 为主）、DELETE 删除（本项目以 POST 为主） | `POST /contract/submit`、`GET /contract/list` |
| 路径参数 | `(:segment)` 占位，CI4 路由 | `GET /contract/detail/(:segment)` |
| 查询参数 | 筛选/分页/排序 | `?page=1&pageSize=20&keyword=xxx` |
| 请求体 | JSON | `{"合同名称": "xxx", "合同金额": 10000}` |
| 响应体 | JSON，统一封装 | `{"code": 0, "message": "success", "data": {...}}` |

### 9.2 统一响应封装

`BaseApiController` 提供统一响应方法：

| 方法 | HTTP 状态 | 响应 code | 用途 |
|---|---|---|---|
| `success($data, $message)` | 200 | 0 | 请求成功 |
| `error($message, $code)` | 200 | 自定义 | 通用错误 |
| `paramError($message)` | 200 | 40001 | 参数校验失败 |
| `notFound($message)` | 200 | 40401 | 资源不存在 |
| `serverError($message)` | 200 | 50001 | 服务器异常 |
| `businessError($message)` | 200 | 42201 | 业务规则校验失败 |

**响应体结构**：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "合同ID": "C2026-0001",
    "合同名称": "采购合同",
    "当前状态": "PENDING"
  },
  "traceId": "abc-123-def",
  "timestamp": "2026-07-23T10:00:00+08:00"
}
```

**约定**：

- HTTP 状态码统一返回 200，业务错误通过响应体 `code` 区分（便于前端统一处理）；
- `traceId` 由 `BaseApiController` 注入，贯穿日志链路；
- `timestamp` 为服务器时间，ISO 8601 格式；
- 例外：OnlyOffice 回调接口响应 `{"error": 0}` 或 `{"error": 1}`（遵循 OnlyOffice 协议）。

### 9.3 X-Server-Trace 头规范

```mermaid
flowchart LR
    A[Controller 入口<br/>t0 = hrtime true] --> B[参数校验<br/>t1]
    B --> C[Service 调用<br/>t2]
    C --> D[SQL 查询<br/>t3]
    D --> E[业务处理<br/>t4]
    E --> F[响应封装<br/>t5]
    F --> G[setServerTrace<br/>计算分段耗时]
    G --> H[X-Server-Trace 头输出]

    subgraph Trace[分段耗时]
        T1[parse 参数校验 t1-t0]
        T2[service Service 调用 t2-t1]
        T3[query SQL 查询 t3-t2]
        T4[business 业务处理 t4-t3]
        T5[encode 响应封装 t5-t4]
        T6[total 总耗时 t5-t0]
    end

    subgraph Slow[慢查询 仅 > 50ms]
        S1[SQL 文本 base64 编码]
        S2[SQL 耗时 ms]
        S3[SQL 行数]
    end

    Trace --> H
    Slow --> H
```

**X-Server-Trace 头格式**：

```
X-Server-Trace: parse=2.1,service=15.3,query=85.2,business=3.1,encode=0.5,total=106.2;slow=1;slow0_sql=<base64>;slow0_time=85.2;slow0_rows=125
```

**规范要点**：

1. **计时精度**：`hrtime(true)` 纳秒精度，`/1e6` 转毫秒，保留 1 位小数；
2. **分段命名**：`parse`/`service`/`query`/`business`/`encode`/`total`；
3. **慢查询阈值**：仅记录 > 50ms 的 SQL，避免响应头溢出；
4. **SQL 编码**：base64 编码，避免特殊字符破坏响应头；
5. **慢查询索引**：`slow=N` 表示 N 条慢查询，`slow0_sql`、`slow1_sql`... 依次列出；
6. **集中实现**：`buildPerformanceTable` 集中在 `BaseApiController`，子类调用 `setServerTrace($segments, $slowQueries)`。

### 9.4 JWT 鉴权流程

```mermaid
sequenceDiagram
    autonumber
    participant U as 用户
    participant F as JwtAuthFilter
    participant S as SessionUserContext
    participant B as Blacklist
    participant C as Controller

    U->>F: 请求 + Authorization: Bearer accessToken
    F->>F: 提取 token
    F->>F: JWT::decode 验签 HS256
    alt 验签失败
        F-->>U: 401 Invalid Token
    else 验签通过
        F->>B: 查询 TokenBlacklist
        alt 在黑名单
            F-->>U: 401 Token Revoked
        else 不在黑名单
            F->>S: 注入 SessionUserContext<br/>userId/roles/部门/属地
            S-->>F: 注入完成
            F->>C: 放行请求
            C-->>F: 响应
            F-->>U: 响应 + X-Server-Trace
        end
    end
```

**双 Token 机制**：

- **accessToken**：短期有效（如 2 小时），每次请求携带；
- **refreshToken**：长期有效（如 7 天），仅用于刷新 accessToken；
- **Token 黑名单**：退出登录、密码修改时 accessToken 加入黑名单，强制失效；
- **签名算法**：HS256，密钥从 `.env` 读取。

### 9.5 错误码规范

| 错误码范围 | 类别 | 示例 |
|---|---|---|
| 0 | 成功 | `0` success |
| 40001-40099 | 参数错误 | `40001` 参数校验失败、`40002` 参数缺失 |
| 40101-40199 | 鉴权错误 | `40101` Token 无效、`40102` Token 过期、`40103` Token 已撤销、`40104` 无权限 |
| 40401-40499 | 资源不存在 | `40401` 合同不存在、`40402` 文档不存在、`40403` 任务不存在 |
| 42201-42299 | 业务规则错误 | `42201` 状态不允许此操作、`42202` 非审批人、`42203` 会签未完成、`42204` 已归档不可编辑 |
| 50001-50099 | 服务器错误 | `50001` 数据库异常、`50002` 文件存储异常、`50003` OnlyOffice 服务不可用 |
| 50301-50399 | 外部依赖错误 | `50301` OnlyOffice 回调失败、`50302` OnlyOffice 下载失败 |

## 10. 安全架构

### 10.1 认证与授权

| 维度 | 机制 | 实现 |
|---|---|---|
| 身份认证 | JWT 双 Token（HS256） | `JwtAuthFilter` + `SessionUserContext` + `TokenBlacklistService` |
| 会话管理 | accessToken 短期 + refreshToken 长期 | 登录接口颁发，刷新接口续期 |
| 路由级授权 | `meta.roles` + 路由守卫 | 前端 `beforeEach` 守卫 |
| 菜单级授权 | 后端动态菜单 | `def_function` + `view_role.功能编码赋权` |
| 按钮级授权 | 8 操作权限映射 | `view_role` 布尔字段 → `contract:add` 等 code |
| 数据级授权 | 行级数据权限 | `AuthorizationService::getDataScope()` 注入 WHERE |
| OnlyOffice 授权 | JWT 双轨制 | 用户 JWT（业务接口）+ OnlyOffice JWT（文档协作） |

### 10.2 SQL 注入防护

| 防护措施 | 实现方式 |
|---|---|
| 参数化查询 | 所有 SQL 通过 `Mcommon::select()/execute()` 执行，参数使用 `?` 占位 + bind |
| 禁止 sprintf 拼接 | 代码审查强制检查，禁止 `$sql = sprintf("... WHERE id = %d", $id)` 模式 |
| 输入校验 | Controller 层对入参类型/范围校验，Service 层二次校验 |
| 字段白名单 | INSERT/UPDATE 仅允许白名单字段，禁止用户输入直接映射字段名 |
| 转义函数 | 动态字段名使用 `Mcommon::quoteIdentifier()` 转义 |
| ORM 优先 | 简单查询使用 CI4 Model 的查询构造器，复杂查询用参数化原生 SQL |

**反面案例（禁止）**：

```php
// 禁止：sprintf 拼接 SQL
$sql = sprintf("SELECT * FROM def_contract WHERE 合同编号 = '%s'", $code);

// 正确：参数化查询
$sql = "SELECT * FROM def_contract WHERE 合同编号 = ?";
$rows = $this->mcommon->select($sql, [$code]);
```

### 10.3 XSS/CSRF 防护

| 防护 | 措施 |
|---|---|
| XSS | Vue 默认转义插值；`v-html` 仅用于受信任内容（如富文本编辑器输出，需后端过滤）；后端对富文本输入使用 HTML 净化库（如 `ezyang/htmlpurifier`） |
| CSRF | CI4 `csrf` Filter 全局启用；OnlyOffice 回调接口 `onlyoffice/callback` 排除 CSRF（走 JWT 验签） |
| Cookie | `HttpOnly` + `Secure` + `SameSite=Strict` |
| CORS | 仅允许前端域名，预检请求 `OPTIONS` 放行 |
| Referer 校验 | 敏感操作（删除/审批）校验 Referer |

### 10.4 敏感信息加密

| 信息 | 存储方式 | 加密算法 |
|---|---|---|
| 用户密码 | `def_user.密码` 字段 | bcrypt（cost=12），禁止 MD5/SHA1 |
| accessToken 签名密钥 | `.env` `JWT_SECRET` | 不存数据库，运维下发 |
| refreshToken | `def_user.refresh_token` 或 Redis | bcrypt 哈希存储，原文不入库 |
| OnlyOffice JWT 密钥 | `.env` `ONLYOFFICE_JWT_SECRET` | 32+ 字节，`openssl rand -base64 48` 生成 |
| 文档下载签名 token | URL 参数 | HS256 签名 + TTL 校验 |
| 数据库密码 | `.env` `database.default.password` | 运维下发，不入版本库 |
| Redis 密码 | `.env` `cache.password` | 运维下发 |

**密钥管理**：

- 所有密钥通过 `.env` 注入，`.env` 不入版本库（`.gitignore` 排除）；
- 生产环境密钥由运维通过环境变量或密钥管理服务下发；
- OnlyOffice JWT 密钥与 Document Server 的 `local.json` 严格一致；
- 密钥轮换建议每 6 个月一次，轮换需停机同步两端。

### 10.5 审计日志

| 日志表 | 用途 | 写入方 | 写入时机 |
|---|---|---|---|
| `sys_sql_log` | 通用 SQL 日志（项目惯例） | `Mcommon::sql_log()` 自动 | 每次 SQL 执行 |
| `def_contract_doc_op_log` | 文档操作留痕（结构化） | `OnlyOfficeCallbackService` + `OnlyOfficeApi` + `ContractDocService` | 打开/编辑/保存/下载/版本创建/回溯/标记/关闭 |
| `def_doc_callback_log` | OnlyOffice 回调原始日志 | `OnlyOfficeCallbackService` | 每次回调 |
| `def_permission_audit` + `def_permission_audit_detail` | 权限变更审计 | `PermissionAuditService` | 角色/用户/菜单/工作流节点赋权变更 |
| `def_workflow_task_log` | 工作流任务流转日志 | `WorkflowTaskService` | 任务提交/同意/拒绝/转签/加签/撤回 |

**日志查询场景**：

### 10.6 OnlyOffice 回调安全

```mermaid
flowchart TD
    A[OnlyOffice 回调请求] --> B[Layer 1: IP 白名单<br/>OnlyOfficeCallbackWhitelist Filter]
    B --> C{IP 在白名单?}
    C -- 否 --> D[403 Forbidden]
    C -- 是 --> E[Layer 2: JWT 验签<br/>OnlyOfficeJwtService::verify]
    E --> F{验签通过?}
    F -- 否 --> G[记录安全日志<br/>401 Unauthorized]
    F -- 是 --> H[Layer 3: 载荷合法性<br/>key/status 必填]
    H --> I{校验通过?}
    I -- 否 --> J[400 Bad Request]
    I -- 是 --> K[Layer 4: 幂等校验<br/>def_doc_callback_log]
    K --> L{已处理?}
    L -- 是 --> M[跳过 返回 error:0]
    L -- 否 --> N[Layer 5: 业务处理<br/>下载/版本/留痕]
    N --> O[返回 error:0]

    P[CSRF 排除<br/>onlyoffice/callback 不走 CSRF] -.-> B
    Q[用户 JWT 排除<br/>onlyoffice/callback 与 download 不走 JwtAuthFilter] -.-> B
```

**五层防护**：

1. **IP 白名单**：`ONLYOFFICE_CALLBACK_WHITELIST` 配置 Document Server 出口 IP/CIDR；
2. **JWT 验签**：OnlyOffice JWT 密钥验签，防伪造回调；
3. **载荷合法性**：`key`、`status` 必填，`status` 取值 0/1/2/3/4/6/7；
4. **幂等校验**：`(文档Key, status, forcesavetype)` 联合唯一索引，防重复处理；
5. **业务校验**：文档归属、版本号单调递增、文件大小/MIME 校验。

**豁免说明**：

- `onlyoffice/callback` 排除 CSRF（OnlyOffice 无法携带 CSRF Token）；
- `onlyoffice/callback` 与 `onlyoffice/download` 排除用户 JWT（由 OnlyOffice JWT 验签替代）；
- 其他 OnlyOffice 接口（`config`、`history`、`history-data`、`oplog`）仍走用户 JWT。

5. 引擎推进时自动通过 `BusinessCallbackRegistry` 回调业务模块；
6. 业务模块在回调中更新自身状态。

### 12.2 OnlyOffice 组件复用

新业务文档接入 OnlyOffice 协作的步骤：

| 步骤 | 动作 | 实现位置 |

| 1 | 在业务表新增文档关联字段或新建文档表 | CI4 Migration |

### 12.4 数据库扩展

CI4 Migration 版本化管理：

| 规范项 | 约定 |
|---|---|
| Migration 目录 | `app/Database/Migrations/` |

| 编号段 | 100001-199999 Workflow、200001-299999 Contract、300001-399999 Permission、400001-499999 OnlyOffice、900001-999999 数据迁移、910001+ 菜单初始化 |

- 生产环境部署时 `php spark migrate` 一键升级；
- 与 Git 提交对应，便于追踪变更历史。

## 14. 技术债务与规避

### 14.1 现有合同模块债务

| 债务 | 现状 | 影响 |
|---|---|---|
| 未分层 | `ContractApi.php` 单控制器硬编码业务逻辑，与 Workbench/Employee 模块惯例不一致 | 难以维护、无法测试、逻辑耦合 |
| SQL 注入风险 | 部分位置 `sprintf` 拼接 SQL | 安全漏洞 |
| 字段命名不一致 | 表结构 `当前流程节点` vs 控制器 `流程节点`；控制器使用 `合同模板ID`/`付款节点` 但表结构未定义 | 字段映射混乱 |
| 未用 Migration | `app/Database/Migrations/` 仅有 `.gitkeep`，手工 SQL 管理 | 变更不可追溯、团队协作冲突 |
| 列表筛选缺失 | `list()` 接口未实现条件筛选，前端搜索 UI 参数被忽略 | 用户体验差 |
| 审批权限缺失 | `approve()` 未校验审批权限，任何登录用户都能审批 | 安全漏洞 |
| 状态机硬编码 | 控制器硬编码 `DRAFT → PENDING → APPROVED → SIGNED → ARCHIVED` | 无法配置多级审批 |
| 已建未用表 | `def_contract_reminder`/`def_contract_party`/`def_contract_attachment`/`def_contract_template` 无对应 CRUD | 资源浪费 |
| 中间态未流转 | 11 种状态枚举中 `APPROVING`/`SIGNING`/`EXECUTING`/`TERMINATED`/`EXPIRED` 未在控制器流转 | 状态机不完整 |

### 14.2 本次重构如何消除债务

| 债务 | 消除措施 | 验收标准 |
|---|---|---|
| 未分层 | 重构为 Controller-Service 分层，`ContractApi` 方法数 ≤ 10，Service 按子域拆分 | 与 Workbench/Employee 模块惯例一致 |
| SQL 注入 | 全部改用 `Mcommon` 参数化查询，禁止 `sprintf` 拼接 | 代码审查零 `sprintf` SQL |
| 字段命名 | 新表 `def_contract` 字段命名规范化，消除 `当前流程节点`/`流程节点` 不一致 | 字段映射表完整 |
| 未用 Migration | 全部新表通过 CI4 Migration 创建，编号段管理 | `php spark migrate` 成功 |
| 列表筛选 | `ContractQueryService::list()` 接收筛选参数，构建 WHERE | 前端搜索 UI 参数被正确接收 |
| 审批权限 | `WorkflowTaskService::approve()` 校验 `current_user == task.处理人` | 非处理人无法审批 |
| 状态机硬编码 | 合同状态由工作流实例状态驱动，不再硬编码 | 状态机可配置 |
| 已建未用表 | 评估复用或归档，新表设计整合相关字段 | 无未用表 |
| 中间态未流转 | 工作流节点配置中间态，状态机完整 | 11 种状态均可流转 |

### 14.3 pnpm 11 严格依赖隔离

**问题**：pnpm 11 默认严格依赖隔离，未在 `dependencies` 声明的包在运行时不可访问，导致构建失败。

**规避措施**：

| 措施 | 实现 |
|---|---|
| `.npmrc` 配置 | `verify-deps-before-run=false` 关闭运行前依赖校验 |
| workspace 配置 | `pnpm-workspace.yaml` 配置 `allowBuilds` 允许构建脚本 |
| 显式声明 | 所有运行时依赖显式声明在 `dependencies`，构建时依赖在 `devDependencies` |
| 锁文件提交 | `pnpm-lock.yaml` 提交版本库，保证团队依赖一致 |
| 不要 hoist | 不配置 `hoist-pattern`，保持严格隔离 |

**`.npmrc` 示例**：

```ini
verify-deps-before-run=false
shamefully-hoist=false
strict-peer-dependencies=false
auto-install-peers=true
```

### 14.4 ag-grid 数值列对齐惯例

**问题**：ag-grid 数值列默认左对齐，与财务/合同金额的右对齐惯例不符；数值筛选器使用文本筛选器导致 `"100" < "20"`（字符串比较）。

**惯例（强制）**：

```typescript
// 合同金额列配置
{
  headerName: '合同金额',
  field: '合同金额',
  type: 'numericColumn',  // 启用数值列类型
  cellStyle: { textAlign: 'right' },  // 右对齐
  filter: 'agNumberColumnFilter',  // 数值筛选器
  comparator: (a: number, b: number) => a - b,  // 数值比较器
  valueFormatter: (params) => 
    params.value?.toLocaleString('zh-CN', { minimumFractionDigits: 2, maximumFractionDigits: 2 }),
  width: 150,
}
```

**规范要点**：

- 数值列必须 `type: 'numericColumn'`；
- `cellStyle: { textAlign: 'right' }` 强制右对齐；
- `filter: 'agNumberColumnFilter'` 使用数值筛选器（支持 `=, >, <, range`）；
- `comparator: (a, b) => a - b` 数值比较器（排序正确）；
- `valueFormatter` 千分位格式化；
- 日期列使用 `agDateColumnFilter` + 日期比较器。

### 14.5 行尾符规范

**问题**：Windows 与 Linux 行尾符不一致导致 Git 提交混乱、shell 脚本无法执行。

**规范（强制）**：

| 文件类型 | 行尾符 | 配置 |
|---|---|---|
| 文本文件（.php/.ts/.vue/.md/.json/.yml/.css/.html） | LF | `.gitattributes` `* text=auto eol=lf` |
| Windows 脚本（.ps1/.bat/.cmd） | CRLF | `.gitattributes` `*.ps1 text eol=crlf`、`*.bat text eol=crlf`、`*.cmd text eol=crlf` |
| 二进制文件（.png/.jpg/.pdf/.docx） | 不转换 | `.gitattributes` `*.png binary`、`*.docx binary` |

**`.gitattributes` 示例**：

```gitattributes
# 默认文本文件 LF
* text=auto eol=lf

# Windows 脚本 CRLF
*.ps1 text eol=crlf
*.bat text eol=crlf
*.cmd text eol=crlf

# 二进制文件
*.png binary
*.jpg binary
*.jpeg binary
*.gif binary
*.ico binary
*.pdf binary
*.docx binary
*.xlsx binary
*.pptx binary
*.zip binary
*.tar.gz binary

# 锁文件
pnpm-lock.yaml text eol=lf
composer.lock text eol=lf
```

**Git 配置**：

```bash
git config core.autocrlf false
git config core.eol lf
git config core.filemode false
```

**执行规范**：

- 新建文件遵循 `.gitattributes` 配置；
- 提交前 `git add --renormalize .` 规整现有文件行尾符；
- IDE 配置默认 LF（VSCode `files.eol: \n`）；
- 禁止在 PR 中混入行尾符变更（除非整体规整）。
