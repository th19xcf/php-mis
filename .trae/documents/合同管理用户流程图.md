## 一、引言

### 1.1 目的

本文档以 Mermaid 流程图与状态图的形式，系统化描述合同管理模块重构升级后各角色用户的操作路径、合同与工作流实例的状态流转、文档协作与版本管理路径、权限审计链路以及异常处理分支。文档作为以下活动的共同参考基准：

- 需求评审：与业务方确认流程是否符合实际审批与协作场景；
- 编码实现：作为后端 `Workflow/Contract/OnlyOffice/Permission` 子域与前端各页面的交互设计依据；
- 测试验证：作为集成测试与 UAT 验收的场景来源；
- 用户培训：作为用户操作手册的配图基础。

### 1.2 范围

本文档覆盖：

1. 合同全生命周期（草稿 → 提交审批 → 多级审批 → 签署 → 归档）；
2. 合同状态机与工作流实例状态机的全部状态与转换；
3. 多级审批中的会签/或签/抄送/转签/加签/撤回分支；
4. OnlyOffice 文档协作、版本管理与操作留痕流程；
5. 权限变更审计与查询导出流程；
6. 审批超时、回调失败、审批人指派为空、版本回溯失败等异常处理；
7. 发起人、审批人、签署人、管理员、查看人五种角色的操作流程。

不在本文档范围：OnlyOffice Document Server 部署、BPMN 拖拽设计器、移动端适配、其他业务模块接入工作流。

### 1.3 图例说明

本文档使用以下 Mermaid 图类型，遵循统一图例约定：

| 图类型 | 用途 | 节点形状约定 |
|---|---|---|
| `flowchart TD` | 业务流程、操作路径 | 圆角矩形为开始/结束；矩形为操作步骤；菱形为判断分支；平行四边形为输入/输出 |
| `stateDiagram-v2` | 状态机 | 圆角矩形为状态；带箭头连线为状态转换，线上标注触发动作 |
| 流程线样式 | 含义 | 实线为正常流转；虚线（`-.->`）为异常或回退分支；粗线（`==>`）为关键主干 |

判断节点统一使用菱形，分支标签（是/否/同意/拒绝等）标注在连线起始处。状态图中的 `[*]` 表示初始状态或终止状态。

## 二、角色定义

合同管理模块共定义五种用户角色，各角色职责与权限边界如下表所示。实际系统中一个用户可同时具备多个角色（多角色合并机制），权限取并集。

| 角色 | 角色编码 | 主要职责 | 核心操作权限 | 数据范围 |
|---|---|---|---|---|
| 发起人 | `initiator` | 发起合同审批流程，维护合同草稿与基础信息，上传/编辑合同文档 | 合同创建、编辑、提交审批、撤回、删除草稿、查看本人发起的合同、文档编辑（草稿态） | 本人发起的合同 |
| 审批人 | `approver` | 处理分配到本人的审批任务，填写审批意见，决定同意/拒绝/转签/加签 | 查看待办任务、查看合同详情、查看/编辑文档（审批态）、同意/拒绝/转签/加签、填写审批意见 | 分配到本人的任务关联合同 |
| 签署人 | `signer` | 对已审核通过的合同执行签署确认，可上传签署附件 | 查看待签署合同、查看文档（只读）、确认签署、上传签署附件、拒绝签署并退回 | 分配到本人的签署任务关联合同 |
| 管理员 | `admin` | 配置工作流定义、管理流程实例、管理权限、查询审计日志、处理异常 | 流程设计器配置、流程实例查询/终止/挂起/恢复、权限赋权变更、权限审计查询/导出、合同全量查询、异常处理 | 全量数据 |
| 查看人 | `viewer` | 只读查看合同信息与文档，无任何写操作权限 | 查看合同列表、查看合同详情、查看文档（只读）、查看版本历史、查看操作留痕 | 授权范围内的合同 |

补充说明：

- 角色与权限通过 `view_role.功能编码赋权` 实现，新增工作流相关权限通过 `def_function` 表扩展；
- 按钮级权限通过前端 `v-if="hasAuth('contract:approve')"` 等指令控制显隐；
- 数据级权限通过 `AuthorizationService` 注入属地/部门/工号过滤条件；
- 管理员默认拥有全部菜单与操作权限，可代行其他角色操作（如代签、代审批），代行操作需在审计日志中标注。

## 三、合同全生命周期流程

本节描述一份合同从创建到归档的完整主干流程，包含拒绝、重新提交、撤回等回退分支。该流程是合同模块的核心业务路径，由合同状态机（第四节）与工作流实例状态机（第五节）共同驱动。

```mermaid
flowchart TD
    Start([开始]) --> Create[发起人创建合同草稿]
    Create --> FillInfo[填写合同基础信息<br/>合同编号/名称/类型/金额/参与方/付款节点]
    FillInfo --> UploadDoc[上传或创建合同文档<br/>关联 OnlyOffice 文件Key]
    UploadDoc --> EditDoc[在线编辑合同文档<br/>协同编辑/版本保存]
    EditDoc --> SaveDraft[保存草稿<br/>合同状态=DRAFT]
    SaveDraft --> DecisionSubmit{是否提交审批?}
    DecisionSubmit -- 否 --> SaveDraft
    DecisionSubmit -- 是 --> Submit[发起人提交审批<br/>启动工作流实例<br/>合同状态=PENDING]
    Submit --> MultiApprove[多级审批流程<br/>见第六节]
    MultiApprove --> DecisionApprove{审批结果?}
    DecisionApprove -- 拒绝 --> Rejected[合同状态=REJECTED<br/>通知发起人]
    Rejected --> DecisionResubmit{是否重新提交?}
    DecisionResubmit -- 是 --> EditDoc
    DecisionResubmit -- 否 --> EndReject([结束-已拒绝])
    DecisionApprove -- 撤回 --> Withdraw[发起人撤回<br/>合同状态回退=DRAFT<br/>工作流实例=已终止]
    Withdraw --> SaveDraft
    DecisionApprove -- 同意 --> Approved[合同状态=APPROVED<br/>工作流实例=已完成]
    Approved --> SignTask[签署任务分配给签署人]
    SignTask --> DecisionSign{签署结果?}
    DecisionSign -- 拒绝签署 --> SignReject[签署人退回<br/>合同状态回退=APPROVED<br/>通知发起人修改]
    SignReject --> EditDoc
    DecisionSign -- 同意签署 --> Signed[合同状态=SIGNED<br/>记录签署时间/签署附件]
    Signed --> Archive[管理员或发起人归档<br/>合同状态=ARCHIVED]
    Archive --> EndArchive([结束-已归档])
```

要点解读：

- 草稿态（DRAFT）允许反复编辑文档与基础信息，可多次保存；
- 提交审批后合同状态由工作流实例驱动，不再由控制器硬编码流转；
- 拒绝与撤回均回退至草稿态，但拒绝由审批人触发、撤回由发起人主动触发（仅当前节点任务未处理时可撤回）；
- 签署阶段为合同状态机与工作流实例解耦环节：审批工作流完成后，签署任务由合同模块单独驱动，避免签署周期过长占用工作流实例；
- 归档为终态，归档后合同只读，版本与留痕仍可查询。

## 四、合同状态机流程图

合同状态机定义合同实体的合法状态与转换规则。重构后合同状态由工作流实例状态与签署/归档动作共同驱动，状态枚举与转换条件如下。

```mermaid
stateDiagram-v2
    [*] --> DRAFT: 创建合同
    DRAFT --> PENDING: 发起人提交审批<br/>启动工作流实例
    PENDING --> DRAFT: 发起人撤回<br/>（当前节点未处理）
    PENDING --> REJECTED: 审批人拒绝<br/>（任一节点拒绝即终止）
    PENDING --> APPROVED: 全部审批节点通过<br/>工作流实例=已完成
    REJECTED --> DRAFT: 发起人重新提交<br/>（编辑后再次提交）
    REJECTED --> [*]: 发起人放弃
    APPROVED --> SIGNED: 签署人确认签署
    APPROVED --> APPROVED: 签署人退回<br/>（保持已审核，等待修改后重签）
    SIGNED --> ARCHIVED: 管理员或发起人归档
    ARCHIVED --> [*]: 终态

    PENDING --> TERMINATED: 管理员终止流程<br/>（异常处理）
    APPROVED --> EXPIRED: 签署超时未处理<br/>（定时任务触发）
    SIGNED --> EXPIRED: 归档超时未处理
    TERMINATED --> DRAFT: 管理员恢复<br/>（重新提交）
    EXPIRED --> APPROVED: 管理员恢复<br/>（重新指派签署人）
```

要点解读：

- 主干状态为 `DRAFT → PENDING → APPROVED → SIGNED → ARCHIVED`，对应合同从草稿到归档的完整生命周期；
- `REJECTED` 为审批拒绝分支，发起人可编辑后重新提交回到 `PENDING`，也可放弃终止；
- `TERMINATED` 为异常终止态，由管理员在异常处理或流程治理时触发，可恢复至 `DRAFT` 重新提交；
- `EXPIRED` 为超时态，由定时任务检测签署或归档环节超时触发，可由管理员恢复并重新指派；
- `ARCHIVED` 为唯一终态，归档后不可再流转，仅支持只读查询；
- 状态转换均需记录操作留痕，权限变更类操作额外写入权限审计表。

## 五、工作流实例状态机

工作流实例是审批流程的运行时载体，每份合同提交审批后创建一个工作流实例。工作流实例状态机独立于合同状态机，但通过 `ContractWorkflowService` 回调桥接二者。

```mermaid
stateDiagram-v2
    [*] --> 待启动: 创建实例<br/>（合同提交时）
    待启动 --> 运行中: 启动实例<br/>（首个节点任务生成）
    运行中 --> 已完成: 全部节点通过<br/>（触发合同 APPROVED）
    运行中 --> 已终止: 任一节点拒绝<br/>（触发合同 REJECTED）
    运行中 --> 已终止: 发起人撤回
    运行中 --> 已终止: 管理员终止
    运行中 --> 已挂起: 管理员挂起<br/>（暂停所有任务）
    已挂起 --> 运行中: 管理员恢复<br/>（继续处理）
    已挂起 --> 已终止: 管理员终止
    已完成 --> [*]: 终态
    已终止 --> [*]: 终态
```

要点解读：

- `待启动` 为瞬时态，合同提交审批时创建实例并立即启动，进入 `运行中`；
- `运行中` 实例下挂多个待办任务，任务处理由 `WorkflowTaskService` 驱动节点推进；
- `已完成` 触发 `ContractWorkflowService::onTaskCompleted()` 回调，合同状态置为 `APPROVED`；
- `已终止` 触发回调，合同状态置为 `REJECTED`（审批拒绝）或回退 `DRAFT`（撤回/管理员终止）；
- `已挂起` 暂停所有待办任务的超时计时，恢复后继续，适用于节假日或外部协调场景；
- 实例状态与任务状态分离：实例 `运行中` 时各节点任务状态独立（待办/已办/已跳过）。

## 六、多级审批流程

多级审批由流程定义（`def_workflow_definition` + `def_workflow_node` + `def_workflow_edge`）配置，运行时由 `WorkflowEngine` 推进。本节展示含会签、或签、抄送、转签、加签分支的完整审批推进流程。

```mermaid
flowchart TD
    Start([合同提交审批]) --> CreateInstance[创建工作流实例<br/>状态=运行中]
    CreateInstance --> ResolveNode1[解析首个审批节点<br/>WorkflowAssigneeResolver]
    ResolveNode1 --> CheckAssignee1{处理人列表<br/>是否为空?}
    CheckAssignee1 -- 是 --> AssigneeEmpty[异常处理<br/>见 11.3]
    CheckAssignee1 -- 否 --> CreateTask1[生成审批任务<br/>def_workflow_task]
    CreateTask1 --> NodeMode1{节点模式?}
    NodeMode1 -- 会签 --> CountersignAll[等待所有任务完成<br/>任一拒绝即终止]
    NodeMode1 -- 或签 --> AnyoneSign[任一任务完成即推进<br/>其余任务标记已跳过]
    NodeMode1 -- 串行 --> SerialWait[等待当前任务完成]
    CountersignAll --> TaskProcess1[审批人处理任务<br/>见第七节]
    AnyoneSign --> TaskProcess1
    SerialWait --> TaskProcess1
    TaskProcess1 --> CcBranch{是否有抄送节点?}
    CcBranch -- 是 --> CcCreate[创建抄送记录<br/>def_workflow_cc]
    CcCreate --> CcNotify[通知抄送人<br/>已读状态跟踪]
    CcNotify --> NextNode1
    CcBranch -- 否 --> NextNode1[评估连线条件<br/>WorkflowConditionEvaluator]
    NextNode1 --> DecisionEnd1{是否到达结束节点?}
    DecisionEnd1 -- 否 --> ResolveNode2[解析下一审批节点]
    ResolveNode2 --> CheckAssignee2{处理人列表<br/>是否为空?}
    CheckAssignee2 -- 是 --> AssigneeEmpty
    CheckAssignee2 -- 否 --> TransferCheck{审批人是否转签?}
    TransferCheck -- 是 --> Transfer[转签给他人<br/>生成新任务<br/>记录任务日志]
    Transfer --> CreateTask1
    TransferCheck -- 否 --> CountersignCheck{审批人是否加签?}
    CountersignCheck -- 是 --> Countersign[加签增加审批人<br/>生成并行任务]
    Countersign --> CreateTask1
    CountersignCheck -- 否 --> CreateTask1
    DecisionEnd1 -- 是 --> Complete[工作流实例=已完成<br/>合同状态=APPROVED]
    Complete --> End([结束])
```

要点解读：

- 节点模式分串行/会签/或签三种，由 `def_workflow_node.会签或签` 字段配置；
- 会签要求所有任务完成才推进，任一拒绝即终止实例；或签任一完成即推进，其余任务自动标记 `已跳过`；
- 抄送节点（`def_workflow_cc`）不阻塞流程，仅通知并跟踪已读状态；
- 转签替换当前任务处理人，加签在当前节点增加并行处理人，二者均生成新任务并记录 `def_workflow_task_log`；
- 连线条件由 `WorkflowConditionEvaluator` 求值，支持基于合同业务字段（金额/类型/属地）的分支路由；
- 任一节点处理人解析为空时进入异常处理流程（见 11.3），实例标记为异常并通知发起人。

## 七、审批人处理任务流程

审批人登录系统后，从"我的待办"入口处理分配到本人的审批任务。本节展示审批人从查看待办到填写意见、做出决策的完整操作路径。

```mermaid
flowchart TD
    Start([审批人登录]) --> TodoList[打开我的待办<br/>workflow-todo/list]
    TodoList --> FilterTask[筛选待办任务<br/>流程类型/发起人/发起时间]
    FilterTask --> SelectTask[选择一条任务<br/>点击行打开处理弹窗]
    SelectTask --> LoadDetail[加载合同详情<br/>基础信息+参与方+金额]
    LoadDetail --> LoadFlow[加载流程进度<br/>节点流转时间线]
    LoadFlow --> LoadDoc[加载合同文档<br/>OnlyOffice 只读或批注模式]
    LoadDoc --> DecisionView{是否需要查看文档?}
    DecisionView -- 是 --> OpenDoc[打开文档编辑器<br/>查看内容/添加批注]
    OpenDoc --> FillOpinion
    DecisionView -- 否 --> FillOpinion[填写审批意见<br/>支持富文本+附件]
    FillOpinion --> DecisionAction{处理动作?}
    DecisionAction -- 同意 --> Approve[提交同意<br/>WorkflowTaskService::taskApprove]
    DecisionAction -- 拒绝 --> Reject[提交拒绝<br/>WorkflowTaskService::taskReject<br/>实例终止]
    DecisionAction -- 转签 --> Transfer[选择转签人<br/>生成新任务]
    DecisionAction -- 加签 --> Countersign[选择加签人<br/>生成并行任务]
    Approve --> CheckAuth{审批权限校验<br/>current_user==处理人?}
    Reject --> CheckAuth
    Transfer --> CheckAuth
    Countersign --> CheckAuth
    CheckAuth -- 失败 --> AuthError[返回无审批权限错误]
    CheckAuth -- 通过 --> WriteOpinion[写入审批意见<br/>def_contract_approval_opinion]
    WriteOpinion --> WriteLog[写入任务流转日志<br/>def_workflow_task_log]
    WriteLog --> AdvanceNode[WorkflowEngine 推进节点]
    AdvanceNode --> Notify[通知发起人/下一审批人]
    Notify --> RefreshTodo[刷新待办列表]
    RefreshTodo --> End([结束])
    AuthError --> End
```

要点解读：

- 审批权限校验是关键安全控制：`taskApprove/taskReject/taskTransfer/taskCountersign` 均校验 `current_user == task.处理人`，否则返回 `businessError('无审批权限')`；
- 审批意见独立存储于 `def_contract_approval_opinion`，支持编辑/查看/追溯/导出，与任务状态解耦；
- 文档查看模式由权限决定：审批人默认只读或批注模式，不可直接修改正文（避免审批人篡改合同内容）；
- 转签与加签均生成新任务并记录日志，转签替换处理人、加签增加并行处理人；
- 任务处理完成后由 `WorkflowEngine` 自动推进节点，无需审批人手动触发下一步。

## 八、文档协作流程

合同文档通过 OnlyOffice 实现多人在线协同编辑。本节展示从打开文档到回调保存、创建版本、记录留痕的完整数据交互流程。

```mermaid
flowchart TD
    Start([用户打开合同文档]) --> AuthCheck{是否有文档查看权限?}
    AuthCheck -- 否 --> Deny[返回无权限错误]
    AuthCheck -- 是 --> RequestConfig[前端请求编辑器配置<br/>onlyoffice/config/docId]
    RequestConfig --> BuildConfig[后端 OnlyOfficeConfigBuilder 构建配置<br/>documentType/document/editorConfig]
    BuildConfig --> SetPermission[根据用户角色设置 permissions<br/>view/edit/download/print/comment/review]
    SetPermission --> SignJwt[OnlyOfficeJwtService 签名<br/>HS256 + ONLYOFFICE_JWT_SECRET]
    SignJwt --> ReturnConfig[返回带 token 的配置]
    ReturnConfig --> LoadApi[前端动态加载 api.js<br/>避免首屏性能影响]
    LoadApi --> InitEditor[初始化 DocsAPI.DocEditor<br/>注入配置]
    InitEditor --> LoadDoc[Document Server 加载文档<br/>从后端 download 接口拉取]
    LoadDoc --> EditMode{编辑模式?}
    EditMode -- 只读 --> ViewOnly[用户只读查看]
    EditMode -- 编辑 --> CoEdit[多人协同编辑<br/>Document Server 内部处理冲突]
    ViewOnly --> StateChange
    CoEdit --> StateChange[onDocumentStateChange 事件<br/>标记编辑中]
    StateChange --> SaveTrigger{触发保存?<br/>status=2 准备保存<br/>status=6 强制保存}
    SaveTrigger -- 是 --> Callback[OnlyOffice 调用回调<br/>onlyoffice/callback]
    Callback --> VerifyJwt[后端验签<br/>Bearer token 或 token 字段]
    VerifyTrigger{验签通过?}
    SaveTrigger -- 否 --> StateChange
    VerifyTrigger -- 否 --> CallbackFail[返回 401<br/>见 11.2 异常处理]
    VerifyTrigger -- 是 --> DownloadDoc[从回调 url 下载文档<br/>存储到 ONLYOFFICE_DOC_STORAGE_PATH]
    DownloadDoc --> CreateVersion[创建版本快照<br/>def_contract_doc_version]
    CreateVersion --> UpdateCurrent[更新文档当前版本<br/>def_contract_doc.当前版本]
    UpdateCurrent --> WriteOpLog[写入操作留痕<br/>def_contract_doc_op_log]
    WriteOpLog --> WriteCallbackLog[写入回调日志<br/>def_doc_callback_log]
    WriteCallbackLog --> ReturnOk[返回 status=0 给 Document Server]
    ReturnOk --> End([结束])
    Deny --> End
    CallbackFail --> End
```

要点解读：

- 编辑器配置由后端构建并 JWT 签名，前端仅负责注入配置，密钥不暴露；
- 权限映射由 `OnlyOfficeConfigBuilder` 根据用户操作级权限动态生成，无编辑权限用户只能查看；
- OnlyOffice 回调状态码：0=未保存、1=编辑中、2=准备保存、3=保存出错、4=无变化关闭、6=强制保存、7=强制保存出错；
- 回调验签失败返回 401，Document Server 会按内置策略重试，重试仍失败进入异常处理流程（见 11.2）；
- 协同编辑冲突由 Document Server 内部处理，应用层仅负责版本快照与留痕，不处理冲突合并；
- 动态加载 `api.js` 避免首屏性能影响，仅在用户打开文档时加载。

## 九、版本管理流程

文档版本管理提供版本历史查看、版本对比、版本回溯、版本标记四项核心能力。本节展示用户在文档编辑器侧边栏操作版本的完整路径。

```mermaid
flowchart TD
    Start([用户打开版本面板]) --> LoadHistory[加载版本历史<br/>onlyoffice/history/docId]
    LoadHistory --> RenderList[渲染版本列表<br/>版本号/创建人/创建时间/版本说明/标记]
    RenderList --> SelectVersion[选择一个版本]
    SelectVersion --> DecisionAction{操作动作?}
    DecisionAction -- 查看 --> FetchData[获取版本数据<br/>onlyoffice/history-data/versionId]
    FetchData --> RenderVersion[在编辑器历史组件中渲染<br/>OnlyOffice 内置 history]
    RenderVersion --> End
    DecisionAction -- 对比 --> SelectTarget[选择目标版本<br/>另一版本号]
    SelectTarget --> CompareVersions[调用版本对比<br/>onlyoffice/history-data 对比两版本]
    CompareVersions --> RenderDiff[渲染差异视图<br/>OnlyOffice 内置对比组件]
    RenderDiff --> End
    DecisionAction -- 回溯 --> ConfirmRollback{确认回溯?<br/>当前版本将被覆盖}
    ConfirmRollback -- 否 --> End([结束])
    ConfirmRollback -- 是 --> CheckPermission{有回溯权限?<br/>管理员或发起人}
    CheckPermission -- 否 --> PermissionDeny[返回无权限错误]
    CheckPermission -- 是 --> BackupCurrent[备份当前版本<br/>创建新版本快照]
    BackupCurrent --> ReplaceDoc[用目标版本替换当前文档<br/>更新 def_contract_doc.当前版本]
    ReplaceDoc --> WriteOpLog[写入操作留痕<br/>操作类型=版本回溯]
    WriteOpLog --> Notify[通知协作者刷新]
    Notify --> End
    DecisionAction -- 标记 --> InputLabel[输入版本标记/说明<br/>如"终版"/"法务审核版"]
    InputLabel --> MarkVersion[调用版本标记接口<br/>contract-doc/version/mark]
    MarkVersion --> UpdateLabel[更新 def_contract_doc_version.版本说明<br/>是否标记=true]
    UpdateLabel --> WriteOpLog
    WriteOpLog --> End
    PermissionDeny --> End
```

要点解读：

- 版本历史由 `def_contract_doc_version` 表存储，每次回调保存（status=2/6）自动创建版本快照；
- 版本对比与查看复用 OnlyOffice 内置 history 组件，无需自研 diff 算法；
- 版本回溯需先备份当前版本，避免误操作丢失数据，回溯失败进入异常处理（见 11.4）；
- 版本标记用于人工标注关键版本（如终版、法务审核版），便于后续检索与追溯；
- 所有版本操作均写入 `def_contract_doc_op_log`，操作类型包括 `版本创建/版本回溯/版本标记`。

## 十、权限审计流程

权限审计捕获角色赋权、用户赋权、菜单赋权、工作流节点赋权四类变更，写入 `def_permission_audit` 与 `def_permission_audit_detail`，支持多维度查询与导出。

```mermaid
flowchart TD
    Start([权限变更操作触发]) --> CaptureBefore[捕获变更前数据<br/>序列化为 JSON]
    CaptureBefore --> ExecuteChange[执行权限变更<br/>view_role/def_role_group/def_function/def_workflow_node]
    ExecuteChange --> CaptureAfter[捕获变更后数据<br/>序列化为 JSON]
    CaptureAfter --> CompareDiff[对比变更前后<br/>计算字段级差异]
    CompareDiff --> WriteAudit[写入 def_permission_audit<br/>变更类型/操作人/操作时间/目标对象/变更前后JSON]
    WriteAudit --> WriteDetail[写入 def_permission_audit_detail<br/>审计ID/字段名/旧值/新值]
    WriteDetail --> InvalidateCache[失效权限缓存<br/>ContextService invalidateConfigCache]
    InvalidateCache --> End1([审计记录完成])

    QueryStart([管理员查询审计]) --> OpenAuditPage[打开权限审计页面<br/>permission-audit/list]
    OpenAuditPage --> Filter[多维度筛选<br/>时间范围/操作人/变更类型/目标对象]
    Filter --> QueryList[调用查询接口<br/>permission-audit/list]
    QueryList --> RenderList[渲染审计列表<br/>ag-grid 展示]
    RenderList --> SelectRecord[选择一条记录查看详情]
    SelectRecord --> QueryDetail[调用详情接口<br/>permission-audit/detail/id]
    QueryDetail --> RenderDiff[渲染变更前后 JSON 差异<br/>diff-match-patch 或自研对比组件]
    RenderDiff --> DecisionExport{是否导出?}
    DecisionExport -- 是 --> Export[调用导出接口<br/>permission-audit/export]
    Export --> GenerateExcel[生成 Excel<br/>复用 ExportService 模式]
    GenerateExcel --> Download[下载 Excel 文件]
    Download --> End2([结束])
    DecisionExport -- 否 --> End2
```

要点解读：

- 审计埋点在现有角色/用户/菜单/工作流节点 CRUD 接口中植入，调用 `PermissionAuditService::record()`；
- 变更前后数据均序列化为 JSON 存储，明细表记录字段级差异，便于精准追溯；
- 权限变更后立即失效 `ContextService` 缓存（TTL 1800s），确保权限实时生效；
- 查询支持按时间范围、操作人、变更类型、目标对象四维度筛选；
- 详情页采用 JSON diff 视图展示变更前后差异，类似 git diff 体验；
- 导出复用 Workbench `ExportService` 模式，输出 Excel 供合规审查归档。

## 十一、异常处理流程

### 11.1 审批超时处理

审批任务超时由定时任务或请求触发 `WorkflowEngine::checkTimeout()` 检测，按节点配置的超时规则自动处理。

```mermaid
flowchart TD
    Start([定时任务或请求触发]) --> ScanTimeout[扫描超时任务<br/>def_workflow_task WHERE 状态=待办 AND 创建时间+超时<现在]
    ScanTimeout --> FindTimeout{发现超时任务?}
    FindTimeout -- 否 --> End([结束])
    FindTimeout -- 是 --> LoadNode[加载节点超时规则<br/>def_workflow_node.超时规则]
    LoadNode --> DecisionRule{超时处理规则?}
    DecisionRule -- 自动同意 --> AutoApprove[自动标记同意<br/>填写意见=超时自动同意]
    DecisionRule -- 自动拒绝 --> AutoReject[自动标记拒绝<br/>实例终止<br/>合同状态=REJECTED]
    DecisionRule -- 仅通知 --> Notify[通知审批人/发起人<br/>不自动处理]
    DecisionRule -- 转交上级 --> TransferUp[转交审批人上级<br/>WorkflowAssigneeResolver 解析上级]
    AutoApprove --> AdvanceNode[WorkflowEngine 推进节点]
    AutoReject --> TerminateInstance[工作流实例=已终止]
    Notify --> End
    TransferUp --> CreateNewTask[生成新任务<br/>记录任务日志]
    CreateNewTask --> End
    AdvanceNode --> End
    TerminateInstance --> End
```

要点解读：

- 超时规则由节点配置决定，支持自动同意/自动拒绝/仅通知/转交上级四种策略；
- 自动处理需填写审批意见（标注"超时自动处理"）并记录任务日志，确保可追溯；
- 转交上级通过 `WorkflowAssigneeResolver` 解析审批人上级，需上级用户存在，否则降级为仅通知；
- 超时检测由定时任务（如每 10 分钟）或请求触发（请求时顺带检测当前实例），避免遗漏。

### 11.2 OnlyOffice 回调失败处理

OnlyOffice 回调验签失败或处理异常时，Document Server 会按内置策略重试，应用层需记录失败并支持人工补救。

```mermaid
flowchart TD
    Start([OnlyOffice 调用回调]) --> VerifyJwt[后端验签]
    VerifyJwt --> DecisionVerify{验签通过?}
    DecisionVerify -- 否 --> Return401[返回 401]
    Return401 --> RetryByServer[Document Server 内置重试<br/>按策略多次重试]
    DecisionVerify -- 是 --> ProcessCallback[处理回调<br/>下载文档/创建版本/写留痕]
    ProcessCallback --> DecisionSuccess{处理成功?}
    DecisionSuccess -- 是 --> ReturnOk[返回 status=0]
    DecisionSuccess -- 否 --> ReturnError[返回非 0 状态码<br/>如 500]
    ReturnError --> RetryByServer
    RetryByServer --> RetryCount{重试次数<br/>是否超限?}
    RetryCount -- 否 --> Start
    RetryCount -- 是 --> WriteFailLog[写入失败日志<br/>def_doc_callback_log 记录失败原因]
    WriteFailLog --> Notify[通知管理员<br/>文档保存失败]
    Notify --> ManualFix{管理员处理}
    ManualFix -- 手动触发 forceSave --> TriggerForceSave[用户手动保存<br/>触发 status=6 强制保存]
    TriggerForceSave --> Start
    ManualFix -- 版本丢失 --> CheckOpLog[查操作留痕<br/>确认最后保存版本]
    CheckOpLog --> RollbackVersion[回溯到最后保存版本<br/>见 11.4]
    RollbackVersion --> End([结束])
    ReturnOk --> End
```

要点解读：

- OnlyOffice 回调失败包括验签失败（401）与处理异常（500），Document Server 均会重试；
- 所有回调（含失败）均写入 `def_doc_callback_log`，记录载荷与处理结果；
- 重试超限后通知管理员，管理员可引导用户手动触发 `forceSave`（status=6）重新保存；
- 若版本丢失，通过操作留痕定位最后成功保存的版本，执行版本回溯恢复。

### 11.3 审批人指派为空处理

`WorkflowAssigneeResolver` 解析节点处理人列表为空时，流程实例无法继续，需标记异常并通知发起人。

```mermaid
flowchart TD
    Start([节点推进时解析审批人]) --> Resolve[WorkflowAssigneeResolver 解析<br/>角色/部门/上级/指定人]
    Resolve --> DecisionEmpty{处理人列表<br/>是否为空?}
    DecisionEmpty -- 否 --> CreateTask[正常生成审批任务]
    CreateTask --> End([结束])
    DecisionEmpty -- 是 --> MarkException[流程实例标记异常<br/>def_workflow_instance.状态=挂起]
    MarkException --> WriteLog[写入任务日志<br/>动作=审批人指派为空]
    WriteLog --> NotifyInitiator[通知发起人<br/>节点X审批人指派为空]
    NotifyInitiator --> NotifyAdmin[通知管理员<br/>需补配审批人或修改流程定义]
    NotifyAdmin --> DecisionFix{处理方式?}
    DecisionFix -- 修改流程定义 --> FixDefinition[管理员修改 def_workflow_node.审批人配置]
    FixDefinition --> ResumeInstance[恢复实例<br/>重新解析审批人]
    ResumeInstance --> Start
    DecisionFix -- 手动指派 --> ManualAssign[管理员手动指派审批人<br/>生成任务]
    ManualAssign --> End
    DecisionFix -- 终止流程 --> Terminate[流程实例=已终止<br/>合同状态=REJECTED]
    Terminate --> End
```

要点解读：

- 审批人指派为空通常因角色/部门下无用户、上级用户不存在、指定人离职等原因；
- 实例标记为 `挂起` 而非直接终止，便于管理员补配后恢复；
- 管理员可通过修改流程定义（影响后续实例）或手动指派（仅影响当前实例）两种方式处理；
- 若无法补配，发起人可放弃并终止流程，合同状态置为 `REJECTED`。

### 11.4 版本回溯失败处理

版本回溯涉及文档替换与版本快照备份，失败时需保证当前版本不丢失，并支持重试。

```mermaid
flowchart TD
    Start([用户触发版本回溯]) --> CheckPermission{有回溯权限?}
    CheckPermission -- 否 --> Deny[返回无权限错误]
    CheckPermission -- 是 --> LoadTarget[加载目标版本文件<br/>def_contract_doc_version]
    LoadTarget --> DecisionExist{目标版本文件<br/>是否存在?}
    DecisionExist -- 否 --> FileMissing[返回版本文件不存在错误<br/>记录留痕]
    DecisionExist -- 是 --> BackupCurrent[备份当前版本<br/>创建新版本快照]
    BackupCurrent --> DecisionBackup{备份成功?}
    DecisionBackup -- 否 --> BackupFail[回溯中止<br/>当前版本未受影响]
    BackupFail --> NotifyFail[通知用户回溯失败]
    BackupFail --> End([结束])
    DecisionBackup -- 是 --> ReplaceDoc[用目标版本替换当前文档<br/>更新 def_contract_doc.当前版本]
    ReplaceDoc --> DecisionReplace{替换成功?}
    DecisionReplace -- 否 --> RestoreBackup[恢复备份<br/>当前版本回滚至备份前]
    RestoreBackup --> WriteFailLog[写入失败留痕<br/>操作类型=版本回溯失败]
    WriteFailLog --> NotifyFail
    DecisionReplace -- 是 --> WriteOpLog[写入操作留痕<br/>操作类型=版本回溯]
    WriteOpLog --> Notify[通知协作者刷新]
    Notify --> End
    FileMissing --> End
    Deny --> End
    NotifyFail --> End
```

要点解读：

- 版本回溯遵循"先备份再替换"原则，确保失败时当前版本可恢复；
- 目标版本文件缺失（如存储清理误删）直接返回错误，不执行回溯；
- 替换失败时自动恢复备份，避免文档损坏；
- 所有回溯操作（含失败）均写入操作留痕，便于追溯与重试。

## 十二、各角色操作流程

### 12.1 发起人操作流程

发起人是合同的创建者与提交者，主要负责草稿维护、提交审批、撤回与重新提交。

```mermaid
flowchart TD
    Login([发起人登录]) --> Menu[进入合同管理菜单]
    Menu --> DecisionEntry{操作入口?}
    DecisionEntry -- 新建合同 --> Create[点击新建按钮<br/>hasAuth contract:add]
    DecisionEntry -- 编辑草稿 --> ListDraft[我发起的列表<br/>筛选状态=DRAFT]
    DecisionEntry -- 查看进度 --> ListAll[我发起的列表<br/>查看流程进度时间线]
    Create --> FillForm[填写合同基础信息<br/>ContractForm.vue]
    ListDraft --> SelectDraft[选择草稿合同]
    SelectDraft --> FillForm
    FillForm --> UploadDoc[上传或创建合同文档]
    UploadDoc --> EditDoc[在线编辑文档<br/>ContractDocEditor.vue]
    EditDoc --> Save[保存草稿<br/>状态=DRAFT]
    Save --> DecisionSubmit{是否提交审批?}
    DecisionSubmit -- 否 --> Save
    DecisionSubmit -- 是 --> Submit[点击提交按钮<br/>hasAuth contract:submit]
    Submit --> StartWorkflow[启动工作流实例<br/>状态=PENDING]
    StartWorkflow --> WaitApprove[等待审批]
    WaitApprove --> DecisionResult{审批结果?}
    DecisionResult -- 拒绝 --> Rejected[收到拒绝通知<br/>查看审批意见]
    Rejected --> DecisionResubmit{是否重新提交?}
    DecisionResubmit -- 是 --> EditDoc
    DecisionResubmit -- 否 --> End([结束])
    DecisionResult -- 撤回 --> Withdraw[主动撤回<br/>hasAuth contract:withdraw<br/>当前节点未处理]
    Withdraw --> Save
    DecisionResult -- 同意 --> Approved[合同状态=APPROVED<br/>等待签署]
    Approved --> End
    ListAll --> End
```

要点解读：

- 发起人权限通过 `hasAuth('contract:add/submit/withdraw')` 控制，按钮按操作级权限显隐；
- 草稿态可反复编辑文档与基础信息，提交后进入审批态，仅撤回可回退；
- 拒绝后可编辑重新提交，文档版本持续累积，审批意见保留可追溯；
- 撤回仅当当前节点任务未处理时可用，避免审批人已处理后被撤回。

### 12.2 审批人操作流程

审批人处理分配到本人的审批任务，操作路径详见第七节，本节补充审批人的整体工作流。

```mermaid
flowchart TD
    Login([审批人登录]) --> Todo[打开我的待办<br/>workflow-todo/list]
    Todo --> Filter[筛选待办<br/>流程类型/发起人/时间]
    Filter --> SelectTask[选择任务]
    SelectTask --> Handle[处理任务<br/>见第七节详图]
    Handle --> DecisionDone{继续处理?}
    DecisionDone -- 是 --> Filter
    DecisionDone -- 否 --> ViewDone[查看已办列表<br/>workflow-todo/done]
    ViewDone --> DecisionOpinion{是否查看历史意见?}
    DecisionOpinion -- 是 --> OpenOpinion[打开审批意见列表<br/>OpinionList.vue]
    OpenOpinion --> EditOpinion{是否编辑意见?}
    EditOpinion -- 是 --> Modify[编辑意见<br/>task/opinion/edit]
    Modify --> ExportOpinion{是否导出?}
    EditOpinion -- 否 --> ExportOpinion
    DecisionOpinion -- 否 --> End([结束])
    ExportOpinion -- 是 --> Export[导出意见<br/>task/opinion/export]
    Export --> End
    ExportOpinion -- 否 --> End
```

要点解读：

- 审批人主要工作集中在"我的待办"，支持待办/已办/抄送/我发起的 Tab 切换；
- 审批意见支持事后编辑与导出，便于归档与合规审查；
- 已办列表可追溯历史处理记录，支持按合同/时间筛选。

### 12.3 签署人操作流程

签署人处理已审核通过（APPROVED）的合同签署任务，可确认签署或退回修改。

```mermaid
flowchart TD
    Login([签署人登录]) --> SignTodo[打开待签署列表<br/>合同状态=APPROVED 且 分配给本人]
    SignTodo --> SelectContract[选择待签署合同]
    SelectContract --> ViewDetail[查看合同详情<br/>基础信息+审批意见]
    ViewDetail --> ViewDoc[查看合同文档<br/>只读模式]
    ViewDoc --> DecisionSign{签署决策?}
    DecisionSign -- 同意签署 --> CheckContent{确认内容无误?}
    CheckContent -- 否 --> ViewDoc
    CheckContent -- 是 --> UploadAttach[上传签署附件<br/>签章扫描件等]
    UploadAttach --> ConfirmSign[点击确认签署<br/>hasAuth contract:sign]
    ConfirmSign --> UpdateStatus[合同状态=SIGNED<br/>记录签署时间/签署人/附件]
    UpdateStatus --> Notify[通知发起人/管理员]
    Notify --> End([结束])
    DecisionSign -- 拒绝签署 --> InputReason[填写退回原因]
    InputReason --> Return[退回发起人<br/>合同状态保持 APPROVED]
    Return --> NotifyReturn[通知发起人修改]
    NotifyReturn --> End
```

要点解读：

- 签署人仅对 APPROVED 状态合同操作，签署任务由合同模块单独驱动（不占用审批工作流实例）；
- 签署人默认只读查看文档，不可修改正文，仅可上传签署附件；
- 拒绝签署不改变合同状态（保持 APPROVED），退回发起人修改后重新进入签署环节；
- 签署完成后合同进入 SIGNED 状态，等待归档。

### 12.4 管理员操作流程

管理员负责流程定义配置、流程实例治理、权限管理、审计查询与异常处理，是系统的运维与治理角色。

```mermaid
flowchart TD
    Login([管理员登录]) --> DecisionModule{操作模块?}
    DecisionModule -- 流程设计器 --> Designer[打开流程设计器<br/>workflow/designer]
    Designer --> DecisionDesign{操作?}
    DecisionDesign -- 新建流程 --> CreateDef[填写流程基本信息<br/>编码/名称/业务类型=合同]
    CreateDef --> AddNode[添加节点<br/>开始/审批/抄送/结束]
    AddNode --> ConfigNode[配置节点<br/>NodeConfig.vue]
    ConfigNode --> ConfigAssignee[配置审批人<br/>AssigneeSelector.vue<br/>角色/部门/上级/指定人]
    ConfigAssignee --> ConfigMode[配置会签或签/超时规则]
    ConfigMode --> ConfigEdge[配置连线条件<br/>EdgeConfig.vue]
    ConfigEdge --> SaveDef[保存流程定义<br/>def_workflow_definition+node+edge]
    SaveDef --> EnableDef[启用流程定义]
    EnableDef --> End([结束])
    DecisionDesign -- 编辑流程 --> EditDef[加载已有定义<br/>版本号+1]
    EditDef --> ConfigNode
    DecisionModule -- 流程实例治理 --> Instance[打开流程实例列表<br/>workflow/instance]
    Instance --> SelectInstance[选择异常实例]
    SelectInstance --> DecisionGovern{治理动作?}
    DecisionGovern -- 终止 --> Terminate[终止实例<br/>合同状态=REJECTED]
    DecisionGovern -- 挂起 --> Suspend[挂起实例<br/>暂停所有任务]
    DecisionGovern -- 恢复 --> Resume[恢复实例<br/>继续处理]
    DecisionGovern -- 查看详情 --> ViewDetail[查看节点流转图+任务+意见]
    DecisionGovern -- 指派审批人 --> ManualAssign[手动指派<br/>见 11.3]
    Terminate --> End
    Suspend --> End
    Resume --> End
    ViewDetail --> End
    ManualAssign --> End
    DecisionModule -- 权限管理 --> Permission[角色/用户/菜单赋权]
    Permission --> ChangeAuth[变更权限<br/>触发权限审计<br/>见第十节]
    ChangeAuth --> End
    DecisionModule -- 权限审计 --> Audit[打开权限审计页面<br/>permission-audit/list]
    Audit --> QueryAudit[查询/导出审计<br/>见第十节]
    QueryAudit --> End
    DecisionModule -- 异常处理 --> Exception[处理异常通知<br/>审批超时/回调失败/指派为空/回溯失败]
    Exception --> HandleException[按第十一节处理]
    HandleException --> End
```

要点解读：

- 流程设计器为表单式配置器（非 BPMN 拖拽），通过垂直节点列表 + 右侧配置面板完成流程定义；
- 流程定义支持版本管理，编辑后版本号递增，旧版本实例继续按旧版本运行，新实例用新版本；
- 流程实例治理支持终止/挂起/恢复/手动指派，覆盖异常处理与流程治理场景；
- 权限变更自动触发审计记录，无需管理员手动操作；
- 异常处理统一收敛到管理员，由第十一节各子流程具体处理。

### 12.5 查看人操作流程

查看人为只读角色，仅可查看授权范围内的合同信息与文档，无任何写操作权限。

```mermaid
flowchart TD
    Login([查看人登录]) --> Menu[进入合同管理菜单<br/>hasAuth contract:view]
    Menu --> List[查看合同列表<br/>数据范围=授权范围]
    List --> Filter[筛选合同<br/>编号/名称/状态/类型/金额/日期]
    Filter --> SelectContract[选择合同]
    SelectContract --> ViewDetail[查看合同详情<br/>基础信息+参与方+金额]
    ViewDetail --> DecisionTab{查看 Tab?}
    DecisionTab -- 流程进度 --> ViewFlow[查看流程时间线<br/>ContractFlowTimeline.vue]
    DecisionTab -- 文档列表 --> ViewDocList[查看文档列表<br/>ContractDocList.vue]
    DecisionTab -- 审批意见 --> ViewOpinion[查看审批意见<br/>OpinionList.vue]
    ViewDocList --> SelectDoc[选择文档]
    SelectDoc --> OpenDoc[打开文档<br/>OnlyOffice 只读模式]
    OpenDoc --> DecisionVersion{查看版本?}
    DecisionVersion -- 是 --> ViewVersion[查看版本历史<br/>ContractDocVersion.vue]
    ViewVersion --> CompareVersion[对比版本<br/>OnlyOffice history 组件]
    CompareVersion --> End([结束])
    DecisionVersion -- 否 --> End
    DecisionTab -- 操作留痕 --> ViewOpLog[查看操作留痕<br/>ContractDocOpLog.vue]
    ViewOpLog --> DecisionExport{是否导出留痕?}
    DecisionExport -- 是 --> CheckExportAuth{有导出权限?<br/>hasAuth contract:export}
    CheckExportAuth -- 是 --> Export[导出留痕 Excel]
    Export --> End
    CheckExportAuth -- 否 --> Deny[无导出权限]
    Deny --> End
    DecisionExport -- 否 --> End
    ViewFlow --> End
    ViewOpinion --> End
```

要点解读：

- 查看人仅持有 `contract:view` 权限，所有写操作按钮（编辑/审批/签署/归档）均隐藏；
- 文档以 OnlyOffice 只读模式打开，permissions 仅含 view，无 edit/download/print；
- 版本对比复用 OnlyOffice history 组件，查看人可对比但不可回溯；
- 操作留痕查看人可见，但导出需额外 `contract:export` 权限，未授权时导出按钮隐藏。
