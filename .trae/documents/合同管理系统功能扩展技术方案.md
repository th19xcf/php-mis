### 2.1 合同电子文档在线修改功能

#### 2.1.1 功能描述
支持对上传的合同文档（Word、PDF、图片）进行在线编辑，实现内容修改、格式调整。

#### 2.1.2 技术方案

**方案A: 集成第三方文档编辑组件（推荐）**

| 组件 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| SmartDoc | 国产、支持Word/Excel/PPT、功能完善 | 商业授权 | ⭐⭐⭐⭐ |
| 永中DCS | 国产、转换稳定 | 功能有限 | ⭐⭐⭐ |
| pdf.js + pdf-lib | 开源、免费 | 仅支持PDF编辑 | ⭐⭐⭐ |
| Office Online | 微软官方、功能最强 | 需Office 365订阅、部署复杂 | ⭐⭐⭐⭐⭐ |

**前端实现**:
```typescript
// 新增组件: ContractDocumentEditor.vue
// 使用 iframe 嵌入第三方文档编辑服务
// 支持: Word在线编辑、PDF预览编辑、手写签名
```

**后端实现**:
```php
// 新增 API 端点
POST   /contract/document/upload    // 上传文档
GET    /contract/document/:id       // 获取文档
POST   /contract/document/:id/edit  // 保存编辑结果
GET    /contract/document/:id/download // 下载文档
```

**数据库变更**:
```sql
-- 新增表: 合同文档版本表
CREATE TABLE `def_contract_document` (
  `GUID` INT NOT NULL AUTO_INCREMENT,
  `合同编号` VARCHAR(50) NOT NULL,
  `文档名称` VARCHAR(200),

  `文件大小` BIGINT,
  `文件类型` VARCHAR(50),
  `版本号` INT DEFAULT 1,
  `状态` VARCHAR(20),  -- editing/completed
  `操作人员` VARCHAR(50),
  `操作时间` DATETIME,
  PRIMARY KEY (`GUID`),
  KEY `idx_合同编号` (`合同编号`)
);

-- 新增表: 合同文档版本历史
CREATE TABLE `def_contract_document_version` (
  `GUID` INT NOT NULL AUTO_INCREMENT,
  `文档GUID` INT NOT NULL,
  `版本号` INT,
  `文件路径` VARCHAR(500),
  `变更内容` TEXT,  -- JSON格式记录变更详情
  `操作人员` VARCHAR(50),
  `操作时间` DATETIME,
```

### 2.2 批注功能

#### 2.2.1 功能描述
允许用户在合同文档的任意位置添加文字批注、回复、编辑与删除。

#### 2.2.2 技术方案

**前端实现**:
```typescript
// 新增组件
ContractAnnotation.vue      // 批注主组件
AnnotationMarker.vue        // 批注标记组件
AnnotationPanel.vue         // 批注列表面板
AnnotationReply.vue         // 批注回复组件
```

**交互流程**:
1. 用户选中文档中的文字/区域
2. 点击"添加批注"按钮
3. 弹出批注输入框
4. 提交后显示批注标记
5. 批注面板显示所有批注列表

**后端API**:
```php
POST   /contract/annotation/create    // 创建批注
GET    /contract/annotation/list      // 获取批注列表
PUT    /contract/annotation/:id      // 更新批注
DELETE /contract/annotation/:id      // 删除批注
POST   /contract/annotation/:id/reply // 添加回复
```

**数据库变更**:
```sql
CREATE TABLE `def_contract_annotation` (
  `GUID` INT NOT NULL AUTO_INCREMENT,
  `合同GUID` INT NOT NULL,
  `文档GUID` INT,
  `父批注GUID` INT DEFAULT NULL,  -- 用于回复
  `批注内容` TEXT NOT NULL,
  `批注位置` JSON,  -- {page: 1, x: 100, y: 200, width: 100, height: 20}
  `批注类型` VARCHAR(20),  -- text/highlight/draw
  `状态` VARCHAR(20),  -- active/resolved/deleted
  `创建人` VARCHAR(50),
  `创建时间` DATETIME,
  `更新时间` DATETIME,
  PRIMARY KEY (`GUID`),
  KEY `idx_合同GUID` (`合同GUID`)
);
```

#### 2.2.3 开发周期预估

| 模块 | 工作量 | 时间 |
|------|--------|------|
| 批注数据模型与API | 中 | 2天 |
| 批注标记组件 | 中 | 3天 |
| 批注面板组件 | 中 | 2天 |
| 批注交互逻辑 | 高 | 3天 |
| 前后端联调 | 中 | 2天 |
| **小计** | - | **12-14天** |

#### 2.2.4 潜在风险

1. **批注定位准确性** - 跨页签批注定位可能存在偏差
2. **性能问题** - 大量批注时渲染性能
3. **并发编辑** - 多用户同时批注的冲突处理

### 2.3 操作留痕功能

#### 2.3.1 功能描述
完整记录所有用户对合同的操作历史，包括修改内容、修改时间、修改人、批注记录等。

#### 2.3.2 技术方案

**前端实现**:
```typescript
// 新增组件
ContractOperationLog.vue  // 操作日志主组件
OperationTimeline.vue     // 时间线展示
OperationDiff.vue         // 变更对比组件（用于文档修改对比）
```

**后端实现**:
```php
// 自动记录所有操作的 AOP 切面
// 切入点: ContractApi 所有写操作
// 日志内容: 操作类型、操作人、操作时间、变更前后数据
```

**数据库变更**:
```sql
CREATE TABLE `def_contract_operation_log` (
  `GUID` INT NOT NULL AUTO_INCREMENT,
  `合同GUID` INT NOT NULL,
  `操作类型` VARCHAR(50),  -- create/update/delete/submit/approve/reject/sign/annotate
  `操作子类型` VARCHAR(50), -- update_field/annotate_add/reply_update等
  `操作人` VARCHAR(50),
  `操作人姓名` VARCHAR(100),
  `操作时间` DATETIME,
  `操作来源` VARCHAR(50),
  `IP地址` VARCHAR(50),
  `变更前数据` JSON,
  `变更后数据` JSON,
  `变更字段列表` JSON,  -- ["合同名称","合同金额"]
  `关联GUID` INT,  -- 关联的批注GUID/签署记录GUID等
  `备注` TEXT,
  PRIMARY KEY (`GUID`),
  KEY `idx_合同GUID` (`合同GUID`),
  KEY `idx_操作时间` (`操作时间`)
);
```

**关键实现点**:

1. **自动记录机制**
   - 后端使用 Hook/Filter 自动拦截所有写操作
   - 记录操作前后数据快照

2. **变更字段追踪**
   - 对比变更前后的 JSON
   - 生成变更字段列表

3. **时间线展示**
   - 按时间倒序展示所有操作
   - 支持按操作类型筛选

4. **变更对比**
   - 文档修改使用 diff 算法对比
   - 支持查看历史版本

#### 2.3.3 开发周期预估

### 2.4 签署流转功能

#### 4.1 功能描述
实现合同的电子化签署流程，支持设置签署顺序、添加签署节点、发送签署通知、在线签署及签署状态跟踪。

#### 4.2 技术方案

**前端实现**:
```typescript
// 新增组件
ContractSigningFlow.vue     // 签署流程主组件
SigningNodeConfig.vue       // 签署节点配置
SigningOrderEditor.vue      // 签署顺序编辑器
SigningNotification.vue     // 签署通知组件
SignaturePad.vue            // 手写签名组件
SigningStatusTracker.vue    // 签署状态跟踪
```

**签署流程设计**:
```
创建签署流程 → 配置签署节点 → 设置签署顺序 → 发送签署通知
     ↓
签署节点1(甲方) → 签署 → 签署节点2(乙方) → 签署 → ... → 完成
```

**后端API**:
```php
POST   /contract/signing/create      // 创建签署流程
GET    /contract/signing/:id         // 获取签署流程详情
PUT    /contract/signing/:id/nodes   // 配置签署节点
POST   /contract/signing/:id/send    // 发送签署通知
POST   /contract/signing/:id/sign    // 执行签署
GET    /contract/signing/:id/status  // 获取签署状态
POST   /contract/signing/:id/cancel  // 取消签署流程
GET    /contract/signing/:id/history // 签署历史
```

**数据库变更**:
```sql
-- 1. 签署流程主表

  `签署方联系人` VARCHAR(100),
  `签署方邮箱` VARCHAR(100),
  `签署方电话` VARCHAR(20),
  `签署顺序` INT,  -- 1=第一签署人,2=第二签署人...
  `节点状态` VARCHAR(20),  -- pending/signed/rejected/expired
  `签署截止时间` DATETIME,
  `签署时间` DATETIME,
  `签署方式` VARCHAR(20),  -- signature_pad/sms_verify/self  手写签名/短信验证/自签署
  `签名图片` VARCHAR(500),
  `签署IP` VARCHAR(50),
  `签署设备` VARCHAR(100),
  `签署意见` TEXT,
  `创建人` VARCHAR(50),
  `创建时间` DATETIME,
  PRIMARY KEY (`GUID`),
  KEY `idx_流程GUID` (`流程GUID`)

-- 3. 签署通知记录表
CREATE TABLE `def_contract_signing_notification` (
  `GUID` INT NOT NULL AUTO_INCREMENT,
  `节点GUID` INT NOT NULL,
  `通知类型` VARCHAR(20),  -- email/sms/in_app
  `通知渠道` VARCHAR(50),
  `通知内容` TEXT,
  `发送状态` VARCHAR(20),  -- pending/sent/failed
  `发送时间` DATETIME,
  `发送人` VARCHAR(50),
  `错误信息` TEXT,
  PRIMARY KEY (`GUID`),
  KEY `idx_节点GUID` (`节点GUID`)

-- 4. 更新现有 def_contract_sign 表以支持更多签署信息
ALTER TABLE `def_contract_sign`

```

## 六、结论

以上四个功能模块在现有架构下**完全可行**，但需要较大的开发投入。建议采用**渐进式开发**策略：

1. **操作留痕**作为基础能力最先实现，可为后续功能提供数据支撑
2. **批注功能**与现有文档查看功能结合，用户体验流畅
3. **签署流转**最为复杂，建议充分调研第三方签署服务
4. **文档在线修改**依赖第三方组件，建议预留集成接口
