## 1. 引言

### 1.1 目的

本说明书对 MIS 系统合同管理模块的系统性升级进行功能层面的完整约定，作为后续系统架构设计、数据库设计、API 接口设计、编码实现、测试验收以及用户手册编写的统一依据。

本次升级的核心目标是：在纠正现有合同模块工程问题（未遵循 Controller-Service 分层、未使用 CI4 Migration、`list()` 未实现筛选、`approve()` 未校验审批权限、字段命名不一致、SQL 注入风险）的基础上，引入通用工作流引擎、OnlyOffice 文档在线协作、文档版本控制、操作留痕、精细化权限审计六大能力，实现合同从起草到归档的全生命周期数字化流转。

本说明书面向以下读者：

- 后端开发人员：依据本说明书进行 Service/Controller 实现；
- 前端开发人员：依据本说明书进行页面与组件实现；
- 测试人员：依据本说明书编写测试用例与 UAT 清单；
- 产品与业务人员：依据本说明书评审功能边界与业务规则；
- 运维人员：依据本说明书了解部署与运行要求。

### 1.2 范围

#### 1.2.1 纳入范围

- 合同全生命周期管理：创建、编辑、删除、提交、审批、签署、归档、列表筛选、详情查看、统计。
- 通用工作流引擎：流程定义、节点配置、连线条件、实例启动、任务处理（同意/拒绝/转签/加签/撤回）、会签或签、抄送、超时处理。
- 审批意见管理：填写、编辑、查看、追溯、导出。
- 合同文档管理：文档创建、OnlyOffice 在线编辑、协同修改。
- 文档版本控制：版本创建、版本对比、版本回溯、版本标记。
- 操作留痕：文档操作记录、查询、导出。
- 权限控制：RBAC、按钮级权限、数据级权限、审批人指派校验。
- 权限变更审计：审计记录、查询、导出。
- 数据迁移：旧表数据迁移到新架构。

#### 1.2.2 不在本次范围

- OnlyOffice Document Server 的部署与运维（由运维另行负责，本系统假设其已就绪）。
- BPMN 可视化拖拽设计器（用户决策采用表单式配置器）。
- 通用工作流引擎接入其他业务模块（本次仅合同作为首个接入方，其他模块未来复用）。
- 现有 Workbench/Employee 模块改造。
- 移动端适配（本次仅 PC 端）。
- 国际化完整覆盖（以中文为主，en-us 按需补充）。

## 2. 系统概述

### 2.1 系统目标

本次升级在 MIS 现有合同模块基础上，构建一套**通用化、可配置、可追溯、可审计**的合同管理能力，达成以下目标：

1. **全生命周期数字化**：合同从起草、审批、签署到归档全过程线上流转，状态由工作流引擎驱动，消除硬编码状态机。
2. **审批流程可配置**：通过表单式流程设计器配置多级审批节点，支持串行、会签、或签、抄送、加签、转签等企业常见审批场景，无需修改代码即可调整流程。
3. **文档在线协作**：集成 OnlyOffice Document Server，支持合同文档多人实时协同编辑，编辑过程留痕可追溯。
4. **版本可追溯**：合同文档每次保存自动生成版本快照，支持版本对比、回溯与标记，确保内容变更可审计。
5. **精细化权限控制**：在现有 RBAC 基础上补全按钮级与数据级权限，审批操作严格校验处理人身份。
6. **权限变更审计**：角色/用户/菜单/工作流节点赋权变更均产生结构化审计记录，可查询可导出。
7. **工程规范对齐**：后端采用 Controller-Service 分层，数据库变更通过 CI4 Migration 版本化管理，字段命名规范化，消除 SQL 注入风险。

### 2.2 用户角色

系统涉及以下用户角色，一个自然人可同时具备多个角色（角色合并机制见 4.6）：

| 角色编码 | 角色名称 | 职责描述 | 典型操作 |
|---|---|---|---|
| R-INITIATOR | 发起人 | 合同起草者，发起审批流程 | 创建/编辑/提交合同、撤回流程、查看进度、管理文档 |
| R-APPROVER | 审批人 | 流程节点配置的处理人，对合同进行审批 | 处理待办（同意/拒绝/转签/加签）、填写审批意见、查看合同详情 |
| R-SIGNER | 签署人 | 合同审批通过后执行签署 | 在线签署、上传签署附件、确认签署完成 |
| R-ADMIN | 管理员 | 系统与流程配置者 | 配置流程定义、管理菜单与权限、查询权限审计、数据迁移 |
| R-VIEWER | 查看人 | 仅具备查看权限的相关方 | 查看合同列表/详情、查看文档、导出（受数据权限约束） |
| R-CC | 抄送人 | 流程节点配置的抄送对象 | 查看抄送记录、标记已读 |

## 3. 功能需求

本章按模块组织功能点，每个功能点包含：功能描述、输入、处理逻辑、输出、业务规则引用。功能点编号规则：`FR-<模块>-<三位序号>`。

### 3.1 合同全生命周期管理（FR-CONTRACT）

本模块覆盖合同从创建到归档的完整生命周期，状态流转由工作流引擎驱动（详见 4.1）。

#### FR-CONTRACT-001 合同创建

| 项 | 内容 |
|---|---|
| 功能描述 | 发起人创建一份新的合同记录，处于草稿状态 |
| 输入 | 合同编号、合同名称、合同类型（字典 `def_contract_type`）、合同金额、币种、签订日期、生效日期、到期日期、合同参与方（甲方/乙方）、合同摘要、附件 |
| 处理逻辑 | 1) 校验必填字段；2) 校验合同编号唯一性；3) 写入 `def_contract` 主表，状态置为 `DRAFT`；4) 关联附件到 `def_contract_attachment`；5) 记录创建人到 `创建人` 字段；6) 写操作后失效合同缓存 |
| 输出 | 新建合同记录的 GUID 与基本信息 |
| 业务规则 | BR-STATE-001（初始状态为 DRAFT）、BR-PERM-001（需 `contract:add` 权限） |
| 异常 | 合同编号重复、必填字段缺失、金额格式非法 |

#### FR-CONTRACT-002 合同编辑

| 项 | 内容 |
|---|---|
| 功能描述 | 发起人修改草稿或被退回的合同内容 |
| 输入 | 合同 GUID + 待修改字段 |
| 处理逻辑 | 1) 校验合同存在；2) 校验当前状态允许编辑（仅 `DRAFT`/`REJECTED` 可编辑）；3) 校验操作人为创建人或具备 `contract:edit` 权限；4) 更新 `def_contract`；5) 失效缓存 |
| 输出 | 更新后的合同信息 |
| 业务规则 | BR-STATE-002（仅 DRAFT/REJECTED 可编辑）、BR-PERM-002 |
| 异常 | 合同不存在、状态不允许编辑、无权限 |

#### FR-CONTRACT-003 合同删除

| 项 | 内容 |
|---|---|
| 功能描述 | 删除草稿状态的合同及其关联数据 |
| 输入 | 合同 GUID |
| 处理逻辑 | 1) 校验合同状态为 `DRAFT`；2) 校验操作人为创建人或具备 `contract:delete` 权限；3) 逻辑删除 `def_contract`（置 `删除标志`）；4) 级联逻辑删除附件、文档记录；5) 失效缓存 |
| 输出 | 删除结果 |
| 业务规则 | BR-STATE-003（仅 DRAFT 可删除）、BR-PERM-003 |
| 异常 | 合同不存在、状态非 DRAFT、无权限 |

#### FR-CONTRACT-004 合同提交审批

| 项 | 内容 |
|---|---|
| 功能描述 | 发起人将草稿合同提交至审批流程，启动工作流实例 |
| 输入 | 合同 GUID、流程定义编码（可选，默认取业务类型对应的启用版本） |
| 处理逻辑 | 1) 校验合同状态为 `DRAFT` 或 `REJECTED`；2) 校验操作人为创建人；3) 调用 `ContractWorkflowService::submit()`；4) `WorkflowInstanceService::start()` 启动流程实例；5) 合同状态置为 `PENDING`，记录工作流实例 ID；6) 通知首节点处理人 |
| 输出 | 工作流实例 ID、首节点任务列表 |
| 业务规则 | BR-STATE-004（DRAFT/REJECTED → PENDING）、BR-WF-001（实例启动）、BR-ASSIGNEE-001 |
| 异常 | 流程定义未启用、审批人指派为空（见 6.5）、合同状态不允许提交 |

#### FR-CONTRACT-005 合同审批

| 项 | 内容 |
|---|---|
| 功能描述 | 审批人处理待办任务，对合同进行同意/拒绝/转签/加签 |
| 输入 | 任务 GUID、处理动作（approve/reject/transfer/countersign）、审批意见、转签目标人/加签目标人（按动作） |
| 处理逻辑 | 1) 校验当前用户为任务处理人；2) 委托 `WorkflowTaskService` 处理；3) 任务完成后由 `WorkflowEngine` 推进节点；4) 触发 `ContractWorkflowService::onTaskCompleted()` 回调更新合同状态；5) 抄送抄送人 |
| 输出 | 处理结果、下一节点任务列表、合同最新状态 |
| 业务规则 | BR-WF-002（任务处理）、BR-COUNTERSIGN-001（会签或签推进）、BR-STATE-005（审批通过/拒绝状态转换）、BR-PERM-004（审批权限校验） |
| 异常 | 非当前处理人、任务已处理、审批意见为空（拒绝时必填） |

#### FR-CONTRACT-006 合同签署

| 项 | 内容 |
|---|---|
| 功能描述 | 审批通过后由签署人在线签署合同 |
| 输入 | 合同 GUID、签署人信息、签署附件（可选）、签署意见 |
| 处理逻辑 | 1) 校验合同状态为 `APPROVED`；2) 校验操作人具备 `contract:sign` 权限且为指派签署人；3) 记录签署信息到 `def_contract_sign`；4) 全部签署节点完成后合同状态置为 `SIGNED` |
| 输出 | 签署结果、合同状态 |
| 业务规则 | BR-STATE-006（APPROVED → SIGNED）、BR-PERM-005 |
| 异常 | 合同状态非 APPROVED、无签署权限、签署人未指派 |

#### FR-CONTRACT-007 合同归档

| 项 | 内容 |
|---|---|
| 功能描述 | 已签署合同归档，进入只读状态 |
| 输入 | 合同 GUID、归档备注 |
| 处理逻辑 | 1) 校验合同状态为 `SIGNED`；2) 校验操作人具备 `contract:archive` 权限；3) 合同状态置为 `ARCHIVED`；4) 归档文档当前版本；5) 失效缓存 |
| 输出 | 归档结果 |
| 业务规则 | BR-STATE-007（SIGNED → ARCHIVED）、BR-PERM-006 |
| 异常 | 合同状态非 SIGNED、无权限 |

#### FR-CONTRACT-008 合同列表查询（含筛选）

| 项 | 内容 |
|---|---|
| 功能描述 | 分页查询合同列表，支持多条件筛选（纠正现有 `list()` 未实现筛选的问题） |
| 输入 | 分页参数（page/pageSize）、筛选参数：合同编号、合同名称（模糊）、合同状态（多选）、合同类型（多选）、金额范围（min/max）、签订日期范围、发起人、参与方 |

## 4. 业务规则

本章独立描述跨功能点通用的业务规则，规则编号 `BR-<域>-<三位序号>`。

### 4.1 合同状态机转换规则（BR-STATE）

合同状态由工作流实例状态驱动，不再硬编码。状态枚举与转换关系如下。

#### BR-STATE-001 初始状态

新建合同状态为 `DRAFT`（草稿）。

#### BR-STATE-002 可编辑状态

仅 `DRAFT` 与 `REJECTED` 状态的合同可被编辑。

#### BR-STATE-003 可删除状态

仅 `DRAFT` 状态的合同可被删除（逻辑删除）。

#### BR-STATE-004 提交审批状态转换

`DRAFT` 或 `REJECTED` 状态的合同提交审批后状态转为 `PENDING`（审批中）。

#### BR-STATE-005 审批结果状态转换

- 工作流实例全部审批节点通过 → 合同状态转为 `APPROVED`（已审批）。
- 任一审批节点拒绝 → 工作流实例终止，合同状态转为 `REJECTED`（已拒绝）。
- 发起人撤回 → 合同状态回退为 `DRAFT`。

#### BR-STATE-006 签署状态转换

`APPROVED` 状态的合同，全部签署节点完成后转为 `SIGNED`（已签署）。

#### BR-STATE-007 归档状态转换

`SIGNED` 状态的合同归档后转为 `ARCHIVED`（已归档），进入只读。

#### 合同状态转换表

| 当前状态 | 目标状态 | 触发动作 | 操作角色 | 前置条件 |
|---|---|---|---|---|
| DRAFT | PENDING | 提交审批 | 发起人 | 工作流实例启动成功 |
| REJECTED | PENDING | 重新提交 | 发起人 | 工作流实例启动成功 |
| PENDING | APPROVED | 审批通过 | 末节点审批人 | 工作流实例完成 |
| PENDING | REJECTED | 审批拒绝 | 任一审批人 | 工作流实例终止 |
| PENDING | DRAFT | 撤回 | 发起人 | 当前节点未处理 |
| APPROVED | SIGNED | 签署完成 | 签署人 | 全部签署节点完成 |
| SIGNED | ARCHIVED | 归档 | 管理员/发起人 | - |
| ARCHIVED | - | 终态 | - | 不可再转换 |

> 注：现有 `def_contract_master` 中 `APPROVING`/`SIGNING`/`EXECUTING`/`TERMINATED`/`EXPIRED` 中间态本次不启用，统一归并到 `PENDING`/`SIGNED`/`REJECTED`/`ARCHIVED`。后续若有需求再扩展。

### 4.2 工作流实例状态机（BR-WF）

#### BR-WF-001 实例启动

流程实例启动时状态为 `运行`，从开始节点推进至首个审批/抄送节点。

#### BR-WF-002 任务处理

- 任务处理前必须校验 `current_user == task.处理人`，否则返回 `businessError('无审批权限')`。
- 任务处理后写入 `def_workflow_task_log`，记录动作（提交/同意/拒绝/转签/加签/撤回/超时）。

#### BR-WF-003 版本管理

同一流程编码可存在多个版本，同时仅一个版本启用。新版本启用时旧版本自动停用。运行中的实例绑定启动时的版本号，不受新版本影响。

#### BR-WF-004 超时规则

节点超时规则配置：超时时长（小时）+ 超时动作（自动同意/自动拒绝/仅通知）。`WorkflowEngine::checkTimeout()` 由定时任务（推荐每小时）或请求触发时调用。

#### BR-WF-005 条件表达式求值

连线条件表达式支持 `field op value` 形式，`op` 支持 `>`/`<`/`>=`/`<=`/`==`/`!=`/`in`。`field` 取自业务变量（如合同金额、合同类型）。无条件的连线默认通过。多条连线同时满足时按配置顺序取第一条。

#### BR-WF-006 转签规则

- 转签目标人必须存在且非当前处理人。
- 转签深度限制由 `Config/Workflow.php` 配置（默认 3 层）。
- 转签后原任务标记为"已转签"，不再可处理。

#### BR-WF-007 加签规则

- 前加签：新任务完成后原任务才可处理。
- 后加签：原任务完成后新任务处理，新任务完成才推进节点。
- 并加签：新任务与原任务并行处理，全部完成才推进（按会签规则）。

#### BR-WF-008 撤回规则

- 仅发起人可撤回。
- 仅当当前节点所有任务均未处理时可撤回。
- 撤回后流程实例状态为 `终止`，合同状态回退为 `DRAFT`。
- 已处理节点的审批人收到撤回通知。

#### BR-WF-009 抄送规则

- 抄送人解析为空时仅记录日志，不阻断流程。
- 抄送记录写入 `def_workflow_cc`，抄送人可标记已读。
- 抄送人无审批权限，仅可查看。

#### BR-WF-010 实例终止

- 管理员可终止任意运行中实例。
- 发起人可终止自己发起的实例（需当前节点未处理，等同于撤回）。
- 终止后未处理任务标记为"已终止"。

#### BR-WF-011 意见编辑规则

- 仅意见填写人可编辑自己的意见。
- 编辑时间窗口默认 24 小时（可配置）。
- 已归档合同的意见不可编辑。
- 编辑后保留历史版本，可追溯。

#### 工作流实例状态转换表

| 当前状态 | 目标状态 | 触发动作 | 操作角色 |
|---|---|---|---|
| - | 运行 | 实例启动 | 系统 |
| 运行 | 完成 | 末节点通过 | 系统 |
| 运行 | 终止 | 拒绝/撤回/管理员终止 | 审批人/发起人/管理员 |
| 运行 | 挂起 | 管理员挂起 | 管理员 |
| 挂起 | 运行 | 管理员恢复 | 管理员 |
| 完成 | - | 终态 | - |
| 终止 | - | 终态 | - |

### 4.3 审批人指派规则（BR-ASSIGNEE）

#### BR-ASSIGNEE-001 审批人类型

节点审批人配置支持四种类型，由 `WorkflowAssigneeResolver` 解析：

| 类型 | 配置格式 | 解析结果 |
|---|---|---|
| 角色 | `{"type":"role","codes":["ROLE_LEGAL","ROLE_FINANCE"]}` | 角色下所有用户列表 |
| 部门 | `{"type":"dept","ids":[1,2]}` | 部门下所有用户列表 |
| 上级 | `{"type":"leader","level":1}` | 发起人指定层级的上级（level=1 直属上级） |
| 指定人 | `{"type":"user","ids":[101,102]}` | 指定用户列表 |

解析结果为空时，按 BR-PERM-004 处理（实例标记异常并通知发起人）。

#### BR-ASSIGNEE-002 多类型组合

同一节点可配置多种审批人类型，解析结果取并集后去重。

### 4.4 会签或签推进规则（BR-COUNTERSIGN）

#### BR-COUNTERSIGN-001 推进规则

| 节点配置 | 推进条件 | 其余任务处理 |
|---|---|---|
| 会签 | 所有任务完成（同意或拒绝） | - |
| 或签 | 任一任务完成 | 其余任务自动标记为"已跳过" |
| 串行（默认） | 单人处理即推进 | - |

会签场景下，任一任务拒绝则节点判定为拒绝（按 BR-WF-006 拒绝处理）。

### 4.5 版本快照规则（BR-VERSION）

#### BR-VERSION-001 forceSave 触发

- OnlyOffice 回调 status=2（准备保存）时创建版本快照。
- 回调 status=6（forceSave）且 `ONLYOFFICE_VERSION_ON_FORCE_SAVE=true` 时创建版本快照。
- 版本号自增，保留所有历史版本。

#### BR-VERSION-002 手动标记

- 用户可对任意版本手动标记（如"定稿版"）。
- 标记不影响版本内容，仅作为标识。

#### BR-VERSION-003 回溯规则

- 版本回溯不删除历史版本，而是以目标版本内容创建新版本。
- 回溯后当前版本指针指向新版本。
- 回溯操作记录留痕。

### 4.6 权限校验规则（BR-PERM）

#### BR-PERM-001 合同创建权限

需具备 `contract:add` 操作权限。

#### BR-PERM-002 合同编辑权限

需具备 `contract:edit` 操作权限，且为合同创建人或具备整表权限。

#### BR-PERM-003 合同删除权限

需具备 `contract:delete` 操作权限，且为合同创建人或具备整表权限。

#### BR-PERM-004 审批权限

- 必须为当前节点任务的指派处理人。
- 不依赖 `contract:approve` 操作权限（审批权限由工作流指派决定）。
- 转签/加签目标人同样需校验身份。

#### BR-PERM-005 签署权限

需具备 `contract:sign` 操作权限，且为合同指派签署人。

#### BR-PERM-006 归档权限

需具备 `contract:archive` 操作权限。

#### BR-PERM-007 数据级权限

- 合同列表/详情/统计/意见查询注入数据权限过滤（属地/部门/工号/整表）。
- 无字段级权限（沿用现有机制）。

#### BR-PERM-008 流程设计权限

需具备 `workflow:design` 操作权限（默认仅管理员）。

#### BR-PERM-009 导出权限

需具备 `contract:export` 操作权限。

#### BR-PERM-010 按钮级权限映射

`view_role` 8 个操作权限映射到按钮 code：

| 操作权限 | 按钮 code | 说明 |
|---|---|---|
| 增加 | `contract:add` / `contract-doc:create` | 新增 |
| 修改 | `contract:edit` | 编辑 |
| 删除 | `contract:delete` | 删除 |
| 导入 | `contract:import` | 导入 |
| 导出 | `contract:export` | 导出 |
| 维护 | `contract:approve` / `contract:sign` / `contract:archive` | 流转操作 |
| 整表 | `contract:all` | 整表权限 |
| 备注 | `contract:remark` | 备注 |

#### BR-PERM-011 文档操作权限

文档操作权限通过 OnlyOffice `permissions` 字段下发：

| 操作 | 权限编码 | permissions 字段 |
|---|---|---|
| 查看 | `contract-doc:view` | `view: true` |
| 编辑 | `contract-doc:edit` | `edit: true` |
| 下载 | `contract-doc:download` | `download: true` |
| 打印 | `contract-doc:print` | `print: true` |
| 评论 | `contract-doc:comment` | `comment: true` |
| 审阅 | `contract-doc:review` | `review: true` |
| 回溯 | `contract-doc:rollback` | - |

#### BR-PERM-012 权限审计查询权限

需具备 `permission-audit:view` 操作权限（默认仅管理员）。

## 5. 非功能需求

### 5.1 性能（NFR-PERF）

#### NFR-PERF-001 列表查询响应时间

合同列表查询（含筛选）在 1 万条数据量下，响应时间 ≤ 1 秒（含数据权限注入）。通过 `X-Server-Trace` 头输出 `buildContext`/`queryTotal`/`queryRecords` 分段耗时定位瓶颈。

#### NFR-PERF-002 详情查询响应时间

合同详情查询（含流程/意见/文档聚合）响应时间 ≤ 800 毫秒。通过 `MetadataCache` 缓存（表指纹校验）降低重复查询开销。

#### NFR-PERF-003 性能计时精度

所有后端方法使用 `hrtime(true)` 计时，纳秒精度，`/1e6` 转毫秒。`buildPerformanceTable` 集中在 `BaseApiController`。

#### NFR-PERF-004 慢查询记录

`X-Server-Trace` 头仅记录 > 50 毫秒的慢查询，SQL 内容 base64 编码避免响应头溢出。

#### NFR-PERF-005 缓存失效

写操作后必须调用 `invalidateConfigCache()` 失效缓存。缓存存储数据 + 表指纹（`information_schema.TABLES.UPDATE_TIME`），读取时校验指纹。

#### NFR-PERF-006 前端首屏性能

OnlyOffice `api.js` 按需动态加载，避免首屏性能影响。ag-grid 数值列显式 `cellStyle: { textAlign: 'right' }` + `agNumberColumnFilter` + 数值比较器。

### 5.2 安全（NFR-SEC）

#### NFR-SEC-001 JWT 用户鉴权

所有业务 API（`contract/*`、`workflow/*`、`workflow-todo/*`、`contract-doc/*`、`permission-audit/*`、`onlyoffice/config/*`、`onlyoffice/history/*`）必须经过 `jwt` 过滤器鉴权。`onlyoffice/callback` 与 `onlyoffice/download` 由 OnlyOffice JWT 验签，排除用户鉴权。

#### NFR-SEC-002 OnlyOffice JWT 签名验签

- 编辑器配置必须由 `OnlyOfficeJwtService::sign()` 签名（HS256，`ONLYOFFICE_JWT_SECRET`）。
- 回调请求从 `Authorization: Bearer <token>` 头或 `token` 字段验签，验签失败返回 401。

#### NFR-SEC-003 SQL 注入防护

- 禁止 `sprintf` 拼接 SQL（纠正现有 `ContractApi.php` 风险）。
- 使用 CI4 Query Builder 或预处理语句。
- 复用 `Mcommon::select()` 等封装方法。

#### NFR-SEC-004 敏感信息脱敏

- `X-Server-Trace` 头 SQL 内容 base64 编码。
- 日志不记录用户密码、Token 等敏感信息。
- OnlyOffice 配置中 `document.url` 不暴露内部存储路径。

#### NFR-SEC-005 审计日志不可篡改

- 操作留痕（`def_contract_doc_op_log`）与权限审计（`def_permission_audit`）仅允许追加写入，禁止更新/删除（数据库层限制）。
- 记录操作人/时间/IP/设备。

#### NFR-SEC-006 输入校验

所有 API 入参必须经过 CI4 Validation 校验，拒绝非法输入。

### 5.3 可用性（NFR-USAB）

#### NFR-USAB-001 暗色模式

前端所有新增页面与组件支持暗色模式（沿用 Soybean Admin 机制）。

#### NFR-USAB-002 国际化

新增文案在 `frontend/src/locales/langs/zh-cn.ts` 与 `en-us.ts` 补充，以中文为主。

#### NFR-USAB-003 错误提示

所有异常返回统一错误码与中文提示，前端通过 NaiveUI message 组件展示。

#### NFR-USAB-004 操作反馈

长时操作（如导出、版本对比）需提供 loading 反馈，避免用户重复点击。

### 5.4 可扩展性（NFR-EXT）

#### NFR-EXT-001 工作流引擎通用性

工作流引擎独立于合同业务（`app/Services/Workflow/`），未来其他业务模块可通过 facade 接入，无需修改引擎代码。

#### NFR-EXT-002 OnlyOffice 编辑器复用

## 6. 异常处理

本章描述各异常场景的处理策略，编号 `EX-<域>-<三位序号>`。

### 6.1 审批超时（EX-WF-001）

| 项 | 内容 |
|---|---|
| 场景 | 审批任务超过节点配置的超时时长未处理 |
| 检测方式 | `WorkflowEngine::checkTimeout()` 由定时任务（每小时）或请求触发时调用 |
| 处理策略 | 按节点超时动作处理：1) 自动同意 → 任务标记为"已超时同意"，推进节点；2) 自动拒绝 → 任务标记为"已超时拒绝"，流程终止；3) 仅通知 → 发送提醒通知给处理人与发起人，不自动处理 |
| 通知 | 站内信/邮件通知处理人与发起人 |
| 日志 | 写入 `def_workflow_task_log`（动作=超时处理） |

### 6.2 文档冲突（EX-DOC-001）

| 项 | 内容 |
|---|---|
| 场景 | 多人协同编辑时的内容冲突 |
| 处理策略 | 由 OnlyOffice Document Server 内部处理协同冲突，应用层不参与。若 Document Server 报告冲突无法自动合并（极少见），回调 status=3（保存出错），应用层记录错误日志并通知参与者重试 |
| 通知 | 站内信通知文档参与者 |
| 日志 | 写入 `def_doc_callback_log` |

### 6.3 OnlyOffice 回调失败（EX-DOC-002）

| 项 | 内容 |
|---|---|
| 场景 | OnlyOffice 回调请求失败（网络异常/验签失败/下载文档失败） |
| 处理策略 | 1) 验签失败返回 401，OnlyOffice 会重试；2) 下载文档失败记录错误日志，返回非 200 让 OnlyOffice 重试；3) 重试上限由 OnlyOffice 配置（默认 3 次）；4) 超过重试上限后人工介入 |
| 日志 | 写入 `def_doc_callback_log`（处理结果=失败） |
| 告警 | 连续失败超过阈值时告警通知运维 |

### 6.4 权限不足（EX-PERM-001）

| 项 | 内容 |
|---|---|
| 场景 | 用户尝试无权限的操作 |
| 处理策略 | 1) API 层返回 403 + 中文提示"无操作权限"；2) 前端按钮层通过 `v-if="hasAuth()"` 隐藏无权限按钮；3) 数据权限不足返回空列表或 403 |
| 日志 | 记录到 `sys_sql_log` 通用日志 |

### 6.5 审批人指派为空（EX-WF-002）

| 项 | 内容 |
|---|---|
| 场景 | 工作流节点审批人配置解析后处理人列表为空 |
| 检测方式 | `WorkflowAssigneeResolver` 解析返回空列表 |
| 处理策略 | 1) 流程实例状态标记为"异常"（挂起）；2) 当前节点任务不创建；3) 站内信通知发起人"审批人指派为空，请联系管理员调整流程配置"；4) 管理员可调整节点审批人配置后恢复实例 |
| 日志 | 写入 `def_workflow_task_log`（动作=指派异常） |

### 6.6 版本回溯失败（EX-DOC-003）

| 项 | 内容 |
|---|---|
| 场景 | 版本回溯时目标版本文件丢失或损坏 |
| 处理策略 | 1) 校验目标版本文件存在且可读；2) 失败时返回错误提示"目标版本文件不可用，请联系管理员"；3) 不修改当前版本；4) 记录错误日志 |
| 日志 | 写入 `def_contract_doc_op_log`（操作类型=版本回溯，详情=失败原因） |

### 6.7 流程定义配置错误（EX-WF-003）

| 项 | 内容 |
|---|---|
| 场景 | 流程定义缺少开始/结束节点、连线形成死循环、条件表达式无法求值 |
| 处理策略 | 1) 保存流程定义时校验结构完整性（必须有且仅有一个开始节点和一个结束节点）；2) 启用时进行模拟运行校验；3) 运行时检测到死循环强制终止实例并告警 |
| 日志 | 写入 `def_workflow_task_log` |

### 6.8 文件大小超限（EX-DOC-004）

| 项 | 内容 |
|---|---|
| 场景 | 上传的合同文档或附件超过大小限制 |
| 处理策略 | 1) 后端校验文件大小（默认 50MB，可配置）；2) 超限返回错误提示"文件大小超过限制"；3) 前端上传组件预校验 |
| 日志 | 无 |

### 7.4 迁移执行与验证

#### 7.4.1 迁移脚本

文件：`e:\code\php\mis\backend\app\Database\Migrations\2026-07-23-900001_MigrateContractData.php`

执行方式：`php spark migrate`

#### 7.4.2 迁移步骤

1. **Dry-run 校验**：脚本支持 `--dry-run` 参数，仅输出迁移计划不实际写入。
2. **创建工作流实例**：为每条合同创建对应的"历史迁移"工作流实例（流程定义编码 `legacy-migration`，状态根据合同状态映射）。
3. **数据迁移**：按字段映射规则迁移合同主表与流程记录。
4. **旧表归档**：旧表重命名为 `def_contract_master_archive`/`def_contract_flow_archive`（或加 `_archive` 后缀），不删除。
5. **迁移日志**：记录迁移开始时间、结束时间、迁移记录数、失败记录数。

#### 7.4.3 迁移验证标准

| 验证项 | 通过标准 |
|---|---|
| 合同主表数据量 | 新表记录数 = 旧表记录数（排除已删除） |
| 合同状态分布 | 新表各状态记录数与映射后旧表一致 |

| 工作流实例 | 每条合同有对应的工作流实例记录 |
| 旧表归档 | 旧表重命名成功，数据可查询 |
| 回滚能力 | 迁移失败时可从归档表恢复 |
