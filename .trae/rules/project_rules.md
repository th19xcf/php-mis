# MIS 项目协作规则

本文件用于记录本项目的协作约定与流程规范，AI 助手在后续会话中应严格遵守。

---

## 一、Git 提交管理流程（mis\backend 与 mis\frontend）

### 1. 适用仓库

- `d:\code\php\mis\backend`（后端，PHP / CodeIgniter 4）
- `d:\code\php\mis\frontend`（前端，Vue 3 + Vite + Soybean Admin）

### 2. 触发机制（严格遵循）

- **仅在人工通过聊天窗口明确提出"提交申请"时**才执行 Git 提交。
- 触发关键词示例（包含但不限于）："提交代码"、"git 提交"、"请提交"、"commit"、"提交一下"、"帮忙提交"等明确表达提交意图的语句。
- **不**进行"是否确认提交"的二次询问。
- **不**设置任何形式的自动提交、定时提交或隐式触发。
- 若用户仅描述修改、要求保存或要求生成 commit 信息但未明确要求"提交"，则只生成信息、暂不执行 `git commit`。
- 两个仓库相互独立，每次提交在各自仓库内独立执行 `git add` 与 `git commit`，生成各自的 commit hash。
- **默认提交范围（关键）**：当用户明确表达提交意图但**未指定仓库**时（例如仅说"提交"、"提交 git"、"commit"），**必须同时分别提交前端和后端两个仓库**，不得只提交其中一侧。
  - 即便用户最近操作的文件只集中在某一侧（如仅改了前端），仍需同时检查另一侧（后端）是否有未提交变更；若有，必须一并提交。
  - 此规则避免出现"前端已提交、后端遗漏"的配对缺失问题（典型场景：新增页面同时包含前端页面与后端 API，仅提交前端会导致接口缺失）。
  - 仅当用户**显式**指定单侧仓库时（如"只提交前端"、"提交 backend"、"后端提交一下"），才单独提交对应仓库。
  - 若检查后某侧仓库无任何未提交变更，跳过该侧并在结果反馈中说明"X 侧无变更，跳过"，无需创建空提交。

### 3. 提交执行流程

当用户明确提出提交申请时，AI 必须按以下顺序执行：

1. **范围确认**：
   - 若用户**显式指定**单侧仓库（如"只提交前端"、"提交 backend"），按指定仓库执行。
   - 若用户**未指定仓库**（仅说"提交"、"提交 git"、"commit" 等），按第 2 节"默认提交范围"规则，**同时分别提交前端和后端两个仓库**。
2. **状态收集（双侧并行）**：对范围内每个仓库分别执行 `git status --porcelain` 与 `git diff --stat` 收集自上次提交以来的所有变更（新增、修改、删除、重命名）。必要时再调用 `git diff` 获取更细粒度的变更摘要。两个仓库的状态收集可并行执行以提升效率。
3. **变更内容整理**：将每个仓库的改动按目录、模块、用途进行归类，确保不遗漏任何文件。前后端涉及同一功能时，应在两个仓库的 commit 信息中相互标注对应 commit 的关联关系（如"对应前端提交：xxxxxxx"）。
4. **生成中文 commit 信息**：基于全部变更生成结构化的中文 commit 信息，要求见下文第 4 节。每个仓库生成独立的 commit 信息。
5. **执行提交**：在每个仓库内独立执行：
   - `git add -A`（或按 `git status` 输出的具体范围添加）
   - `git commit -m "<生成的中文信息>"`
   - 输出提交结果（commit hash、变更统计）。
   - 两个仓库的提交顺序建议：先后端、后前端（保证前端调用方在 commit 时后端 API 已就位）；若两侧无依赖关系可任意顺序。
6. **结果反馈**：向用户报告每个仓库的提交是否成功、涉及哪些文件，并附上最终的 commit 信息内容。若某侧无变更，明确说明"X 侧无变更，跳过"。

### 4. Commit 信息规范

- **必须使用中文**编写，禁止英文 commit。
- **首行（标题）**：简洁概括本次提交的核心目的，建议格式 `<类型>: <简要说明>`，例如 `feat: 新增员工批量导入功能`、`fix: 修复登录验证码失效问题`、`refactor: 重构工作台查询服务`、`docs: 补充接口文档` 等。类型可参考：feat / fix / refactor / perf / style / docs / test / chore / build / ci。
- **正文**：使用列表或小节形式，**完整覆盖所有变更点**，包含但不限于：
  - 新增 / 修改 / 删除的文件及对应模块；
  - 关键逻辑调整点（接口、配置、数据结构、权限、UI 等）；
  - 与上下游模块的影响说明；
  - 必要的兼容性、迁移或回滚提示。
- **范围粒度**：若一次提交涵盖多个不相关模块，应在正文中分小节描述（如"## 后端变更"、"## 前端变更"），保证信息一目了然。
- **不**留空泛描述（如"修复 bug"、"更新代码"），必须具体到功能点。

### 5. 不允许的行为

- 未经用户明确要求，不得执行 `git commit`、`git push`、`git reset --hard`、`git checkout .`、`git clean -f` 等写操作命令。
- 不得伪造或猜测未发生的变更内容。
- 不得在 commit 信息中出现与本次实际变更无关的内容。
- 不得在用户未要求的情况下自动整理 / 合并 / 修改历史 commit。

### 6. 辅助脚本

- 仓库根目录已存在 `git-commit.ps1`，仅作为本地辅助脚本参考。AI 助手在执行提交时仍需按本规则第 3、4 节生成符合要求的中文 commit 信息，**不得**直接复用脚本中可能存在的英文或模板化信息。

## 二、其他约定

- 后端与前端目录下的 `.env`、`writable/`、`logs/`、`vendor/`、`node_modules/`、`builds/` 等敏感或构建产物文件不应纳入提交范围。
- 每次提交后应在 `.trae/documents/` 中的"实施进展"文档追加对应的进展条目（如适用）。
